## はじめに：あなたのサイトはまだ「20年前のプロトコル」を使っていないか？

HTTP/1.1 は1997年に生まれた。Googleがまだ存在せず、スマートフォンはなく、Webページの平均サイズは今の1%にも満たなかった時代のプロトコルだ。

HTTP/2 は2015年に多重化をもたらし、アプリ層の Head-of-Line Blocking を解決した。大きな前進だったが、致命的な盲点があった――問題が「下へ押し込まれた」だけで、根本的には解決されていなかった。

**HTTP/3 は根本からの再設計だ。** 下層プロトコルを TCP から UDP ベースの QUIC に入れ替え、ネットワーク接続の仕組みを根本から変えた。

2025年10月時点で、HTTP/3 のグローバル採用率はすでに **35%**（Cloudflare 調べ）。「将来の技術」ではなく、今まさに進行中だ。

---

## なぜ HTTP/2 では不十分なのか？

HTTP/2 はアプリ層のキューイング問題を多重化で解決した。同一接続で複数リクエストを並列処理できるように見えた。しかし――

**問題がトランスポート層に潜伏したままだった。**

TCP は順序保証のバイトストリームだ。1つのパケットが失われると、後続のすべてのパケットは、それが全く別の HTTP ストリームのものであっても、再送を待たなければならない。パケットロスが多い環境では、HTTP/1.1 で複数 TCP 接続を張る「力技」が HTTP/2 の単一接続より速いというケースすら起きた。

---

## HTTP/3 の本質的変革：TCP を捨て QUIC を採用

HTTP/3 は TCP を完全に捨て、**UDP** ベースの **QUIC プロトコル**（RFC 9000）を採用した。

### 特徴①：ストリーム単位のパケットロス回復

各 QUIC ストリームが独立してロス回復を処理する。画像リクエストのパケットが失われても、JavaScript ファイルのロードはブロックされない。これが本当の意味での多重化だ。

### 特徴②：0-RTT 接続再開

- **初回アクセス**：1-RTT ハンドシェイク（TCP+TLS の 2〜3 RTT より大幅に速い）
- **再アクセス**：0-RTT。最初のパケットに HTTP リクエストを同梱し、サーバーが即座に応答

翌日ユーザーがサイトを開いた瞬間、ハンドシェイク待ちなしでファーストビューのレンダリングが始まる。

### 特徴③：接続マイグレーション

従来の TCP 接続は 4 タプル（送信元 IP・ポート＋宛先 IP・ポート）に紐付けられている。WiFi からモバイル回線に切り替わった瞬間、IP が変わり接続が切断、ハンドシェイクをやり直すことになる。

QUIC は **Connection ID** で接続を識別するため、IP が変わっても接続が維持される。モバイルユーザーの体験向上に直結する。

### 特徴④：TLS 1.3 内蔵

QUIC は暗号化をトランスポート層に組み込んでいるため、独立した TLS ハンドシェイクが不要。安全かつ高速。

## 実測パフォーマンスデータ

RequestMetrics のベンチマーク（Minnesota → 複数データセンター）：

| サイト種別 | リソース規模 | HTTP/2 → HTTP/3 改善率 |
|-----------|------------|----------------------|
| 小規模サイト | 600KB・20リソース | 15〜20% |
| コンテンツサイト | 10MB・105リソース | 25〜35% |
| SPA アプリ | 15MB・115リソース | 30〜40% |

**重要：ネットワーク品質が悪いほど HTTP/3 の優位性が大きい**

| ネットワーク状況 | 改善率 |
|----------------|-------|
| 良好（<1% パケットロス） | 緩やかな改善 |
| 中程度のロス（2〜5%） | 30%+ 高速化 |
| 高ロス/モバイル弱電界（10%+） | **60%+** 高速化 |

同一ページで HTTP/1.1 が3秒、HTTP/2 が1.5秒、HTTP/3 が **0.8秒**。47% の改善は宣伝数字ではなく実測値だ。

## 2分で完了：HTTP/3 を有効にする

### Nginx（v1.25.0+）

```nginx
server {
    listen 443 ssl;
    listen 443 quic reuseport;  # 重要：UDP リスニング

    http2 on;
    http3 on;

    server_name example.com;
    ssl_protocols TLSv1.3;          # HTTP/3 は TLS 1.3 必須
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    ssl_early_data on;    # 0-RTT 有効化
    quic_retry on;        # DDoS 対策
    quic_gso on;          # パフォーマンス最適化（カーネル要件あり）

    # ブラウザへ HTTP/3 対応を通知（必須！）
    add_header Alt-Svc 'h3=":443"; ma=86400' always;
}
```

**よくある落とし穴3つ：**

1. **ファイアウォールで UDP 443 を開放する**――設定したのに動かない場合はほぼこれ：
   ```bash
   ufw allow 443/udp
   ```

2. **Nginx が http_v3_module をビルドしているか確認：**
   ```bash
   nginx -V 2>&1 | grep http_v3_module
   ```

3. **`Alt-Svc` ヘッダーを忘れない**――ブラウザはこれで HTTP/3 対応を検出する。ないと永遠に HTTP/2 が使われる

### Caddy（最もシンプルな方法）

設定の手間を省きたいなら Caddy を使おう――HTTP/3 はデフォルト有効、ゼロ設定：

```caddy
example.com {
    root * /var/www/html
    file_server
    # HTTP/3 は自動的に有効、追加設定不要
}
```

Caddyfile 10 行で HTTP/3 + HTTPS 自動証明書が完結する。個人プロジェクトや中小規模サービスで運用コストを最小にしたい場合の最適解だ。

## ブラウザ互換性：気にしなくていい

2026年3月時点で、Chrome、Firefox、Safari、Edge はすべて HTTP/3 に対応済み。

未対応ブラウザ（現在はほぼ存在しない）は `Alt-Svc` ヘッダーのネゴシエーションで自動的に HTTP/2 にフォールバックする。ユーザーへの影響はゼロだ。

---

## 今すぐ移行しなくていい場面

**イントラネットサービス**：192.168.x.x のような内部サービスは元々レイテンシが低く、弱電界最適化や 0-RTT の恩恵が限定的。急ぐ必要はない。

**0-RTT のセキュリティ注意点**：`ssl_early_data` を有効にすると、POST リクエストにリプレイアタックのリスクが生じる。べき等でない操作には `Early-Data` ヘッダーの処理が必要。

---

## 確認方法：HTTP/3 が本当に動いているか？

Chrome DevTools → Network パネル → 列ヘッダーを右クリック → Protocol 列を追加。`h3` と表示されれば成功。

コマンドラインでも確認できる：

```bash
curl -I --http3 https://example.com 2>&1 | head -5
```

## 結論：35%は転換点を超えた数字だ

グローバルトラフィックの35%が HTTP/3 を使っている。「使うかどうか」という問いはもう終わっていて、「いつ移行するか」の段階に入っている。

Nginx なら UDP 設定の追加だけ。Caddy ならデフォルトで有効。これほどデプロイコストが低いアップグレードは珍しい。外部公開サービス、特にモバイルユーザー向けには、30〜60% のパフォーマンス改善が Core Web Vitals スコアとユーザー体験に直接影響する。

競合がすでに有効にしているかもしれない。
