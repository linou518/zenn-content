## 背景

社内向けのECサイトを構築していたとき、ユーザー登録フォームに最初から「メールアドレス必須」の制約を入れていた。理由は単純で、「認証といえばメール」という思い込みがあったから。

ところが実際に運用してみると、問題が起きた。**既存ユーザーの多くは電話番号しか持っていない**のだ。具体的には、LINE経由で登録されたユーザーは電話番号だけをDBに持っており、メールアドレスは未設定。そのユーザーが新規登録しようとすると「Bad Request」で弾かれていた。

## 問題の構造

フロントエンド（React）は電話番号・パスワードで送信。バックエンド（Node.js/TypeScript）はその電話番号でDBを引いてユーザーを検索するが、ここで「メールが必須フィールドとして未設定」のユーザーが正常に処理されていなかった。

さらに、「新規登録」として電話番号を送ったとき、すでにDBに同じ電話番号が存在するのに「新規作成」を試みてしまい、Unique制約違反でコケるケースもあった。

まとめると2つの問題：
1. 登録時：emailを必須フィールドにしていたためバリデーションで弾かれる
2. 登録時：既存ユーザーと同じ電話番号で「新規作成」を試みてUniqueエラー

## 解決アプローチ

### フロントエンドの変更

メールアドレスフィールドを「任意（optional）」に変更。`required` 属性を外し、プレースホルダーも「任意」表記に。

```tsx
<input
  type="email"
  placeholder="メールアドレス（任意）"
  value={email}
  onChange={(e) => setEmail(e.target.value)}
/>
```

### バックエンドの変更

ログイン・登録のロジックを「phone OR email どちらでもOK」に統合。

```typescript
// Before: emailで検索して失敗
const user = await User.findOne({ where: { email } });

// After: phone または email で検索
const user = await User.findOne({
  where: {
    [Op.or]: [
      phone ? { phone } : null,
      email ? { email } : null,
    ].filter(Boolean),
  },
});
```

さらに「電話番号がすでにDBに存在する場合は新規作成をスキップしてログイン処理に流す」ロジックを追加した。

```typescript
if (existingUser) {
  // 既存ユーザー → パスワード検証してログイン
  return handleLogin(existingUser, password);
} else {
  // 新規ユーザー → 新規作成
  return handleRegister({ phone, email, password });
}
```

## 学んだこと

**「メール＝認証の基本単位」という前提を疑う**ことが最初に必要だった。特にアジア・東南アジアのユーザーにとっては、電話番号のほうが圧倒的に主要な識別子になっているケースが多い。LINEやWhatsAppベースのビジネスであればなおさら。

もう一つは、**「登録」と「ログイン」を明確に分けすぎると詰まる**という設計の話。新規登録APIと既存ログインAPIが完全に分離されていると、「電話番号がすでに存在するが本人は気づいていない」ときにどちらのエンドポイントを使えばいいかわからなくなる。今回のように「両方試して存在確認してからログインに流す」という設計が現実には合っていることがある。

## まとめ

- ユーザーの実態に合わせて認証フィールドを「OR」にする
- フロントの必須バリデーションとバックエンドの検証を常に一致させる
- 「既存ユーザーの新規登録エラー」は設計の前提ミスのサインである

小さなバグ修正に見えて、裏にあったのは「ユーザーとはどういう存在か」の設計ミスだった。ちゃんと動くコードより、現実に合っているコードのほうが大事、という当たり前のことを改めて実感した。
