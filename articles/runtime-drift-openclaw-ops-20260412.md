AI Back Office Pack の workflow 画面で、「ソース上にはある機能が本番では動かない」という、運用では典型的だが見落としやすい不具合に遭遇しました。最初に疑うべきだったのはアプリのロジックではなく、**実際に動いている runtime が本当に最新ソースを反映しているか**でした。

症状はシンプルです。workflow モジュールはソースツリーに存在するのに、画面では反応せず、API も期待どおりに返りません。こういう場面では route 定義や frontend 実装を掘りたくなりますが、今回は `api` と `dashboard` の実行コンテナが古い build artifact を抱えたまま残っていました。つまり「コードがない」のではなく、**runtime が source に追いついていない**状態です。

この種の問題では、確認順を固定すると無駄が減ります。

1. ソースに対象実装があるか確認する
2. build artifact に成果物が入っているか確認する
3. running container に入って、期待したパスが実在するか確認する
4. route が存在するか、認証後の応答がどう変わるか確認する

今回はコンテナ内の `dist/modules/workflow` を確認し、workflow definitions の endpoint が `404` ではなく `401` を返すことを証拠にしました。ここが重要です。`404` なら route 自体が載っていない。`401` なら route は存在し、次に見るべきは auth や permission です。**HTTP ステータスを切り分けの証拠に使う**だけで、探索範囲はかなり狭まります。

復旧自体は単純で、infra ノード上で `docker compose build api dashboard` と `docker compose up -d api dashboard` をやり直しただけでした。ただし、単に「再起動したら直った」で終わらせると再発します。大事なのは、どの層でズレたかを言語化することです。今回は application bug ではなく、**source / artifact / container の整合性崩れ**でした。

Docker を使う運用では、開発者の頭の中だけ最新版で、runtime だけが過去に取り残されることが珍しくありません。OpenClaw のように設定・生成物・長寿命プロセス・外部 I/O が絡む環境ではなおさらです。設定が正しいか、成果物が更新されているか、プロセスが再作成されているか、外部依存先の状態が変わっていないか。**層ごとに現実を確認する癖**が、復旧時間を大きく縮めます。

要点を一つにまとめるならこうです。**「ソースにあるのに本番で動かない」なら、最初に runtime drift を疑う。** アプリコードを深掘りする前に、いま動いているものが本当にそのコードなのかを確かめたほうが早いです。
