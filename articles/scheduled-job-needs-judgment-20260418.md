# cronは繰り返すだけでは弱い。定期ジョブに必要なのは判断力

多くのチームは cron を「決まった時刻にコマンドを実行する仕組み」として扱います。これは間違いではありません。ただし半分しか合っていません。**本当に安定した定期ジョブは、繰り返すだけでなく、実行時に判断できる必要があります。**

毎日同じシェルを機械的に走らせるだけのジョブは、現実の入力変化に弱いです。ある日は素材があり、ある日はありません。外部 API が正常な日もあれば、レート制限に当たる日もあります。主経路で終わる日もあれば、代替経路に入るべき日もあります。

だから私は cron job を、**時間で起動する固定動作**ではなく、**時間で起動する意思決定の入口**として設計した方が強いと考えています。

ブログ自動配信を例にすると、要求は一見単純です。毎日 21:00 に記事を出す。しかし実際の実行では、少なくとも次の分岐があります。

1. 当日の下書きがある → 校正して翻訳し、各プラットフォームへ配信する
2. 下書きがない → その場でテーマを選び、1本書いてから配信する
3. 一部のプラットフォームだけ失敗した → ほかは続行し、最後に結果を構造化して残す

この種の仕事を「毎日同じコマンドを打つだけ」で作ると、運用はすぐ脆くなります。

## 信頼できないのは時刻ではなく入力側

自動化の失敗は、trigger が動かなかったからではなく、**入力が常に存在すると決め打ちしていた**ことから始まる場合が多いです。

典型的な思い込みはこうです。

- 当日の素材は必ず用意されている
- 認証トークンは失効しない
- API のレスポンス形式は変わらない
- 投稿先は一時的に詰まらない
- 前回実行の状態は今回に影響しない

このうち1つでも外れると、そのジョブは自動化ではなく、毎日障害を作る装置になります。

cron 設計で重要なのは expression よりも、入力判定と分岐戦略です。

## 良い定期ジョブが先に答えるべき3つの問い

## 1. 主経路に入れるだけの入力が今あるか

最初にやるべきことは、業務処理ではなく入力確認です。

たとえばブログ配信なら、まず当日の `YYYY-MM-DD_*.md` が存在するかを見るべきです。存在すれば編集・配信経路へ、なければ fallback の執筆経路へ進む。この分岐があるだけで、「素材がないから今日の 21:00 を丸ごと無駄にした」という事故を避けられます。

## 2. 主経路が詰まったときの降格経路は何か

降格は、失敗時に warning を1行出すことではありません。設計段階で決めておくべきです。

- 今日はスキップする
- 代替コンテンツを生成する
- 一部の媒体だけ先に出す
- 生成物だけ保存して外部公開しない
- 人手確認へ切り替える

降格経路のない自動化は、アラーム付き手作業に近いです。

## 3. 終了後に第三者が何を確認できるか

cron の本質的な弱点は、実行中に誰も見ていないことです。

だから結果は、最初から監査可能な形で残す必要があります。

- 何を入力として確認したか
- どの分岐に入ったか
- どの媒体が成功し、どこが失敗したか
- 失敗は HTTP ステータスか、権限か、形式不一致か
- 最終成果物はどこに置いたか

サマリのない定期ジョブは、成功していても運用しづらいです。

## cron を単なる script launcher から当番エージェントへ変える

私が勧めるのは、cron が直接複雑な shell を抱えるのではなく、**小さな当番エージェントを起こす**構成です。

違いは明確です。

- script launcher は、決め打ちの手順しか実行できない
- 当番エージェントは、文脈を確認し、分岐を選び、結果を整理できる

この設計は、次のような日常タスクと相性が良いです。

- ブログ配信
- データ集計
- 週報生成
- 受信箱トリアージ
- ヘルスチェック
- 定期クリーンアップ

共通点は1つです。**起動時刻は固定でも、その日の正しい動きは固定ではない。**

この前提を受け入れるなら、タスク構造は「定時 + コマンド」ではなく、「定時 + 判定 + 分岐 + 報告」にするべきです。

## ひとつの判断基準

私は今、定期ジョブの出来を次の問いで見ています。

**今日の入力が昨日と違っていても、このジョブは違う行動を取りつつ、なお妥当な結果を返せるか。**

答えが No なら、そのジョブはまだ単純反復の段階です。

cron の価値は、時刻どおりに起きることではありません。起きたあとに、その日の現実を判断できることにあります。
