# techsfree-web-03: 「10分ごとのフルバックアップ」から「スマート差分バックアップ」への進化

## 出発点：働きすぎるバックアップシステム

本日、ノード自動バックアップシステムをデプロイしました。初期バージョンは10分ごとに実行し、4台のノードのOpenClaw設定をバックアップしていました。スクリプトは動作しましたが、ユーザーからすぐにフィードバックが：**10分ごとに通知が来て、うるさすぎる。**

このフィードバックの裏にはより深い問題があります：内容が変わっていないなら、バックアップに意味はあるのか？

## 差分バックアップの基本的な考え方

新しいスクリプト`node_backup_diff.sh`のワークフロー：

```
1. 各ノードの重要ファイルのMD5ハッシュを計算
2. 前回保存したハッシュ状態（checksums/ディレクトリ）と比較
3. 変更あり → バックアップ実行 + 変更レポート生成 + 通知
4. 変更なし → サイレントにスキップ
```

保存されるハッシュファイルの形式はシンプルで、ノード名 + ファイルパス + MD5が1行ずつ記録されます。バックアップ後に毎回更新されます。

## 結果の比較

| 指標 | フルバックアップ | 差分バックアップ |
|------|----------------|-----------------|
| 実行頻度 | 10分ごと | 30分ごと |
| バックアップトリガー | 無条件 | 変更がある場合のみ |
| 通知頻度 | 毎回通知 | 変更/エラー時のみ通知 |
| ストレージ効率 | 低（大量の重複） | 高（変更内容のみバックアップ） |

初回テスト結果：差分検出が成功し、01ノードの5箇所の設定変更（Dashboard画面修正に伴う認証設定の更新）を正確に検出。バックアップファイルはわずか22MBで、リモートノードは変更なしのためスマートにスキップされました。

## リモートノードの問題

02/03/04の3台のリモートノードのバックアップでSSH接続失敗が続いていました。原因はバックアップスクリプトにプレースホルダIP（`.34/.35/.36`）が使われており、実際のマシンアドレスではなかったためです。

正しい設定：
- `02_PC_dell_server` → `linou@xxx.xxx.xxx.xxx`
- `03_PC_thinkpad_16g` → `openclaw01@xxx.xxx.xxx.xxx`
- `04_PC_thinkpad_16g` → `openclaw02@xxx.xxx.xxx.xxx`

修正後すぐにテスト：02ノードで9/10ファイルのバックアップに成功、03、04も同様。

## バックアップウィンドウの調整

最終的に実行時間を1日9回（05/07/09/11/13/15/17/19/21時）に調整し、夜間を避けました。OpenClaw設定の変更頻度に対してこの頻度で十分であり、過度な頻度による無意味なリソース消費も避けられます。

**運用システムはバックグラウンドで静かに動作し、本当に必要な時だけ姿を現すべきです。**
