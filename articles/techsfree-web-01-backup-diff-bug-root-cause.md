# ノード自動バックアップの差分検出がなぜ機能しなかったのか——根本原因分析

## 問題の症状

システムは2時間ごとに4つのノードに対して自動バックアップを実行する。設計上、ファイルに変更があった場合のみバックアップを実行する——差分検出（diffing）は、ストレージとI/Oを節約するための核心メカニズムだ。

しかし、監視で異常なパターンが発見された：

```
07:02  02_node バックアップ完了（9/10 ファイル）
09:02  02_node バックアップ完了（9/10 ファイル）
11:00  02_node バックアップ完了（9/10 ファイル）
13:00  02_node バックアップ完了（9/10 ファイル）
15:00  02_node バックアップ完了（9/10 ファイル）
...
```

バックアップのたびに、02、03、04ノードは「変更あり」と報告し、ファイル数が完全に一致していた（9/10、9/10、8/10）。01ノード（ローカル）のみ正常——変更なしのときはスキップした。

これはおかしい。2時間ごとに各ノードで完全に同じ数のファイルが変更されることはあり得ない。

## 根本原因の特定

差分検出の核心データファイル `file_hashes.json` を確認したところ、重大な問題が発見された：

```json
{
  "02_node": {
    "/path/to/config.json": "MISSING",
    "/path/to/agents.json": "MISSING",
    ...
  }
}
```

リモートノードのすべてのハッシュ値が `MISSING` だった。

つまり：バックアップ時に、システムはリモートファイルの内容を読み取り、ハッシュを計算し、`file_hashes.json` の記録と比較する——記録が `MISSING` なので「ファイルが変更された」と判断し、バックアップを実行する。バックアップ完了後、**ハッシュの書き戻しロジックが失敗するか、そもそも存在しない**ため、次回の検出時も記録は依然として `MISSING` で、無限ループとなる。

差分検出のロジックチェーンの断裂点：

```
リモートファイル読み取り → ハッシュ計算 → 旧ハッシュと比較（MISSING）→ 変更と判定 → バックアップ実行
                                                                                  ↓
                                                                     [ここで新ハッシュを書き込むべき]
                                                                     [しかし書き込みが失敗または欠如]
```

## 影響

| 指標 | 期待される動作 | 実際の動作 |
|------|------------|---------|
| バックアップトリガー条件 | ファイル変更時 | 毎回トリガー |
| リモートノードのバックアップ頻度 | 必要時（低頻度） | 2時間ごとに強制フル |
| ストレージ使用量 | 増分積み上げ | 各サイクルで線形増加 |
| 差分検出の効果 | 不要なバックアップをフィルタリング | 完全に機能しない |

ローカルノード01_PC_dell_serverは影響を受けない。ローカルファイルの読み取りパスが異なり、ハッシュの書き戻しロジックが完全であるためだ。これがバグ特定の鍵となった——ローカルは正常、リモートは全滅、問題はリモートファイル処理の分岐にある。

## 修正方向

核心となる修正点：バックアップ完了後に、今回計算したハッシュ値を `file_hashes.json` に書き込み、`MISSING` を置き換える。

疑似コード：

```python
# バックアップ後
new_hashes = compute_hashes(remote_files)
update_hash_store(node_id, new_hashes)  # この一行が以前は実行されていなかった
```

検証方法：修正後、最初のバックアップサイクル終了後に `file_hashes.json` を確認——`MISSING` が実際のハッシュ値に置き換わっていれば修正有効。次のサイクルで変更のないノードは「スキップ」と表示されるはず。

## 教訓

**差分検出のテストは「変更があるときにバックアップをトリガーする」だけでなく、「変更がないときにバックアップをスキップする」も検証すべきだ。**

今回のバグの根本原因は、後者が一度も検証されていなかったことだ。システムは表面上「動いていた」——バックアップタスクは定刻に起動し、ファイルはコピーされ、ログは成功を示していた——しかし差分メカニズムはすでに静かに機能不全に陥っていた。この種の「機能しているように見えるが論理は断裂している」バグは、期待頻度と実際の頻度を比較して初めて発見できる。

運用ツールの正確性は機能の豊富さより重要だ。毎回「変更あり」と言う差分検出は、ないよりも悪い。
