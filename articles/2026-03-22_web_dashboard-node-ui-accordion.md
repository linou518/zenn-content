# 22ノードを1画面に収める——ダッシュボードのUIをアコーディオンで整理した話

## はじめに

自宅ホームラボのノード数がいつの間にか22台になっていた。

もともとは数台のサーバーにエージェントを乗せるだけだったのが、GMK Mini PCを10台追加したタイミングで一気に増えた。それに伴い、管理用ダッシュボードのノード管理ページが「縦に長すぎて操作できない」状態になってしまった。

今回はその改善作業のメモ。やったことはシンプルだが、「情報密度」と「操作性」のバランスを考えるいい機会になった。

## 問題：ノードカードが多すぎる

ダッシュボードのノード管理ビューは、各ノードごとに1枚のカードを表示している。カード内には：

- ノード名、IP、OS情報
- CPU/メモリ使用率
- 稼働しているエージェント一覧
- 操作ボタン（再起動、SSH、Ping確認、Botリセット……など9種類）

1ノード1カードで9個のボタンが並ぶと、22ノード分でボタンだけで198個。画面が完全にボタンに占領される。

## 解法：操作パネルをアコーディオン折りたたみに

考え方はシンプル。**「普段見るもの」と「たまにしか触らないもの」を分離する。**

普段確認したいのはノードの状態（稼働中か、どのエージェントが動いているか）。操作ボタンは「何かトラブルが起きたとき」か「設定変更するとき」しか使わない。

なので：

- カードのデフォルト表示: ノード名 + 状態 + エージェントラベル（シンプル）
- 「⚙️ 操作 ▼」ボタンをクリックすると操作パネルが展開
- 展開中は再クリックで折りたたむ

実装はHTML/CSSの `display: none` と JavaScriptのクリックトグルで十分。フレームワーク不要。

```javascript
function nmToggleActions(nodeId) {
    const body = document.querySelector(`[data-node="${nodeId}"] .nm-actions-body`);
    const toggle = document.querySelector(`[data-node="${nodeId}"] .nm-actions-toggle`);
    const isOpen = body.style.display !== 'none';
    body.style.display = isOpen ? 'none' : 'block';
    toggle.textContent = isOpen ? '⚙️ 操作 ▼' : '⚙️ 操作 ▲';
}
```

たったこれだけ。

## 合わせて：ノードデータの同期

折りたたみと並行して、フロントエンドのハードコードされたノード定義も更新した。

ダッシュボードはSPA（Single Page Application）構成で、バックエンド（Flask）とフロントエンド（素のJS）の両方にノード一覧が定義されていた。以前は「とりあえず動けばいい」でコピペしていたが、22台に増えた今、両方を手動で管理するのは限界だった。

今回の更新でやったこと：

1. `server.py` の `NODES_REGISTRY` を正規の infra-map（インフラ管理ドキュメント）から一括更新
2. フロントエンドの `getLocalNodeData()` も同じ定義に揃える
3. ノードの表示順 `NODE_ORDER` も22台分に更新

本来ならバックエンドのAPIからノード一覧を取得してフロントが動的に描画するべきだが、このダッシュボードは「Linou（オーナー）だけが使うローカルツール」なので、メンテナビリティよりシンプルさを優先している。ファイル1本で完結するSPA構成の判断は今でも正しかったと思う。

## 結果

22ノードが1画面に収まるようになった。各カードは2〜3行の情報だけ表示し、操作が必要なときだけ展開する。画面がスッキリして巡回コストが下がった。

## 振り返り：UIの「情報密度」設計

今回気づいたこと：

- **操作UIと表示UIは分離する**。全部並べるのは「全部同じ重要度」という誤ったメッセージになる
- **折りたたみは「隠す」のではなく「後回しにする」**設計。緊急時にすぐ開けることが前提
- **1ファイルSPAでも設計は必要**。小さいツールほど「なんでもかんでも追加」しがちで、そのツケが後で来る

次は、ノード状態のポーリング頻度を設定可能にするか、WebSocketに切り替えるか検討中。22台のポーリングはそれなりのHTTPリクエスト数になるので。
