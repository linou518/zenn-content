月末締め切り当日に自動化スクリプトが壊れているのに気づく——これほど最悪なシナリオはない。それを防ぐために「週次dry-run」を仕込んだ話を書く。

---

## 背景：月末自動化の全体像

TechsFreeでは毎月末に以下の一連の作業を自動化している：

| 時刻 | 作業 |
|------|------|
| 17:00 | Hajimari勤怠システム自動入力（Playwright） |
| 18:00 | Identity 作業報告書＋請求書 Google Forms自動提出（CDP） |
| 19:00 | エッジテク・anyenv 請求書メール送信プレビュー |

これらは会社の売上・支払いに直結する業務だ。手動でやると毎月2〜3時間かかっていたものを、スクリプト化して全自動で回している。

技術スタックはざっくりこんな感じ：

- **Playwright** + Chrome CDP → 各Webサービスのブラウザ自動操作
- **freee API** → 領収書OCR・取引登録・請求書データ取得
- **Python / Node.js** → 各種自動化スクリプト（計9本）
- **SSH** → 内部サーバーへのリモート実行
- **OpenClaw cronジョブ** → 毎月最終営業日の自動トリガー

---

## 問題：壊れていても月末まで気づかない

この構成の致命的な弱点がある。**各コンポーネントは「月に一度しか使われない」**のだ。

- freeeトークン（OAuth）は有効期限があり、知らないうちに失効する
- シンボリックリンクが切れていても、参照されるまでエラーにならない
- Chrome CDPポートが何かの拍子に落ちていても、誰も気づかない
- スクリプトがリファクタリングの過程で移動されても、古いパスは残ったまま

月に一度しか走らないシステムの「動作保証」をどうするか。本番実行まで信頼性がわからないのでは困る。

---

## 解決策：毎週月曜朝に「ドライラン」を走らせる

専用のエージェント（`techsfree_monthly`）が毎週月曜10:00に自動チェックを実行し、結果を内部メッセージバス経由でJoeに報告する仕組みを作った。

チェック項目は以下の通り：

```
1. SSH接続確認（内部サーバーへの疎通）
2. Chrome CDP起動確認（ポート19222へのWebSocket接続）
3. Playwright動作確認（バージョン確認 + ブラウザ起動）
4. freeeトークン確認（.freee_token.json の存在・読み込み可否・有効期限）
5. スクリプト存在確認（9本すべてのパスチェック）
6. .envファイル確認（必須キーの存在チェック）
```

「実際の操作は一切行わない」のがポイントだ。ログインしない、フォームに入力しない、APIを叩かない。依存関係のヘルスチェックだけに徹している。

---

## 実際に捕まえた事例：2026-03-16

このdry-runが実際に価値を発揮した瞬間があった。

3月16日（月）の週次チェック、報告内容：

```
✅ SSH接続: OK
✅ Chrome CDP (19222): OK
✅ Playwright v1.58.2: OK
⚠️ freeeトークン: 異常
   .freee_token.json がシンボリックリンク切れ
   → リンク先 /path/to/.freee_token.json が存在しない
✅ スクリプト全件(9本): OK
```

freeeトークンのシンボリックリンクが切れていた。

何が起きていたかというと、以前の整理作業でディレクトリ構成を変えた際に、`config/` 配下にトークンファイル本体が移動されていたのに、symlink先として古いパスが残ったままになっていた。

**修正内容：**

1. トークンディレクトリを再作成
2. トークンファイルをコピー（NFS上はsymlinkが不安定なためコピーで対応）
3. スクリプト側の `TOKEN_PATH` を直接 `config/` を参照するよう修正

月末まで2週間あった。余裕を持って修正できた。

もしこれを月末当日に発見していたら、手動でトークンを探してパスを修正して再実行して……という作業を締め切り直前に行うことになっていた。

## 本日（2026-03-23）の結果

今朝のdry-run報告：

```
✅ SSH接続: OK
✅ Chrome CDP (19222): OK
✅ Playwright v1.58.2: OK
✅ freeeトークン: OK (.freee_token.json 全キーあり)
✅ スクリプト全件: OK (9本)
結論: 全正常、月末作業に支障なし
```

3月末まで約1週間。システムは正常稼働中。

## 設計の要点

**なぜ毎週月曜か**

月末自動化は月に一度しか走らない。でも依存関係（トークン、接続、ファイル）は毎日変わりうる。週次チェックであれば問題発覚から月末まで少なくとも数日の猶予が確保できる。

**なぜ別エージェントか**

月末自動化の実行エージェント（techsfree_monthly）にdry-runを兼務させている。同じエージェントが本番実行時の環境チェックもdry-runの健全性チェックも担う。コードの重複がない。

**なぜメッセージバス経由の報告か**

異常があればJoeに即座に通知が届く。無視できない。「どこかにログがある」ではなく「毎週月曜朝に必ずインボックスに届く」形にすることで、確実に目に触れる。

---

## パターンとして使える

月末自動化に限らず、「低頻度・高重要度」の自動化全般に使えるパターンだと思う。

- 年次更新処理（税務申告、各種契約更新）
- 四半期レポート生成
- 定期バックアップのリストアテスト

「本番を走らせるその日まで動いているかわからない」を「毎週OK確認済み」に変える。コストは軽量なヘルスチェックだけ。
