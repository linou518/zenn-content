## TL;DR

OpenClawで20以上のAIエージェントを運用する中で、「全エージェント横並び」では管理が破綻する。解決策として導入したのが**マネージャーAgent + 専門Agent**の2層構造。本記事では、6つのパーソナルエージェント（学習・投資・健康・生活・不動産・学習支援）を1人のマネージャー（Jack）で統括する設計と運用の実際を紹介する。

## 背景：なぜフラット構造は破綻するか

20以上のエージェントが同一階層に並ぶと、以下の問題が発生する：

1. **人間の認知負荷** — 誰に何を頼んだか覚えきれない
2. **リソース競合** — 複数エージェントが同時にCPU/メモリ/APIを奪い合う
3. **コンテキスト漏洩** — 仕事の話と投資の話が混ざる
4. **優先度不明** — 緊急の健康アラートと日常の学習タスクが同列

人間の組織と同じで、10人以上の直属部下を持つマネージャーは機能不全に陥る。

## 設計：2層アーキテクチャ

```
Linou (人間)
  ├── Joe (総管・メインアシスタント) ← 仕事・インフラ・全体統括
  │     ├── work-a, work-b, hobby, family, personal ...
  │     └── 基盤サービス（消息バス、OCM、監視）
  └── Jack (パーソナルマネージャー) ← 個人生活・学習・資産
        ├── learning (学習管理)
        ├── xuesi (学習支援・中国語)
        ├── investment (投資分析)
        ├── health (健康管理)
        ├── life (生活管理)
        └── real-estate (不動産)
```

### なぜ2人のマネージャーか

- **Joe**: 仕事・インフラ・技術的な管理全般。24時間稼働のcronジョブ、サーバー監視、開発タスク。
- **Jack**: パーソナル領域。投資判断、健康トラッキング、学習計画、生活のあれこれ。

この分離の最大の利点は**コンテキストの分離**。Joeのセッションに投資ポートフォリオの話が入り込まないし、Jackのセッションにサーバー障害対応が混ざらない。

## マネージャーAgentの3つの役割

### 1. 優先度の判定

6つのエージェントが同時に動くとき、何を優先するか。Jackが判定する：

```
緊急度マトリクス:
  🔴 即座対応: 健康アラート（health）、期限付き契約（real-estate）
  🟡 当日中: 投資の売買判断（investment）、重要な学習期限（learning）
  🟢 通常: 日常の学習進捗（xuesi）、生活の買い物リスト（life）
```

実装上は、各エージェントが消息バス経由でJackにタスクを報告し、Jackが優先度に応じてリソース（モデルの選択、実行タイミング）を調整する。

### 2. エージェント間の連携調整

単独では解決できないタスクがある：

- **投資判断 + 不動産**: 「今の資産配分で不動産購入は可能か？」→ investmentとreal-estateの両方のデータが必要
- **健康 + 生活**: 「今週の運動計画に合わせた食事メニュー」→ healthとlifeの連携
- **学習 + 学習支援**: 「技術書の読書計画を中国語学習と統合」→ learningとxuesiの同期

Jackがハブとなって情報を中継し、矛盾する指示が出ないように調整する。

### 3. 報告の集約

6つのエージェントがそれぞれ人間に報告するとノイズになる。Jackが集約して1つのサマリーにする：

```
📊 今日のパーソナルサマリー:
- 📚 learning: JLPT N1模試完了、正答率78%（前回+5%）
- 💹 investment: 日経平均-2.1%、保有銘柄影響軽微、アクション不要
- 🏃 health: ランニング5km完了、週間目標達成率80%
- 🏠 real-estate: 新着物件2件（条件一致度：高1/中1）
```

## 実装の要点

### 消息バスによる通信

エージェント間通信はHTTP APIベースの消息バス（message bus）を使用：

```bash
# 各エージェント → Jack へ報告
curl -X POST http://internal-server:8091/api/send \
  -H "X-Bus-Token: <TOKEN>" \
  -d '{"from":"investment","to":"jack","subject":"日次レポート","body":"..."}'

# Jack → 各エージェント へ指示
curl -X POST http://internal-server:8091/api/send \
  -d '{"from":"jack","to":"health","subject":"優先度変更","body":"今週は運動量を減らして休養優先"}'
```

### GOALS.mdによる自律的タスク管理

各エージェントは自分の`GOALS.md`を持ち、Jackがheartbeatごとに確認・更新する：

```markdown
# GOALS.md - investment
## 🔴 今日やること
- [ ] 米国株マーケットレビュー（開場前）
- [ ] ポートフォリオリバランス検討

## 🟡 今週やること  
- [ ] Q1決算レビュー対象銘柄リストアップ
```

Jackは全エージェントのGOALS.mdを横断的に見て、リソース競合や依存関係を調整する。

### セッション分離のメリット

各エージェントが独自のセッション（≒独自のコンテキストウィンドウ）を持つことで：

- **専門知識の蓄積**: investmentのセッションには金融データが、healthには運動記録が蓄積される
- **compactionの影響軽減**: 1つのセッションが圧縮されても他に影響しない
- **並列実行**: 独立したタスクは本当に並列で動く

## 運用で学んだこと

### やってよかったこと

1. **マネージャーの判断基準を明文化** — SOUL.mdに「果断に決定する、委員会を開かない」と書いた。AIは明示しないと合議を始める。
2. **報告フォーマットの標準化** — 各エージェントの報告形式を統一し、Jackが処理しやすくした。
3. **エスカレーション経路の明確化** — 技術的問題はJoe、パーソナル問題はJack、両方に関わるものはJoe経由Linou。

### 失敗したこと

1. **最初はマネージャーなしで運用** — 6つのエージェントが人間に直接報告→情報洪水で機能不全。
2. **マイクロマネジメントの誘惑** — 全エージェントの行動を逐一確認→マネージャーのセッションが肥大化。「管理はするが管理しすぎない」方針に転換。
3. **リソース競合の未対策** — 全エージェントが同時にOpusモデルを要求→API rate limit。モデル選択の優先度ルールを導入。

## まとめ

AIエージェントの階層管理は、人間の組織設計と驚くほど似ている：

- **スパン・オブ・コントロール** — 1人のマネージャーが見られるのは5-7人
- **コンテキストの分離** — 関心事の異なるチームは分ける
- **判断の委譲** — 全部トップが決めると詰まる
- **報告の集約** — 情報は上がるほど圧縮される

OpenClawのようなmulti-agent基盤を使いこなすには、エージェントの数を増やすだけでなく、**管理構造を設計する**ことが不可欠。フラットな構造は5エージェントまで。それ以上なら、マネージャーを置こう。
