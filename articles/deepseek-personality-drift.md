# LLM Fallback Chain の落とし穴：DeepSeek で人格が消えた日

## はじめに

Multi-agent システムを運用していると、LLM プロバイダーのレート制限やクォータ切れは日常茶飯事だ。だから fallback chain を組む——メインモデルがダメなら次、それもダメならその次。合理的な設計のはずだった。

ある日、その「合理的な設計」が想定外の障害を引き起こした。**モデルが切り替わった瞬間、AI エージェントの人格が消えた。**

## 何が起きたか

うちの OpenClaw クラスタでは、複数のノードに20以上のエージェントが稼働している。各エージェントには `SOUL.md` という人格定義ファイルがあり、口調・性格・専門性・禁止事項まで細かく記述されている。

この日、Claude Sonnet のクォータが100%に到達した。設定上の fallback chain はこうなっていた：

```
primary: anthropic/claude-sonnet-4-6
fallbacks:
  - openai/gpt-4o-mini
  - deepseek/deepseek-chat
```

Sonnet がクォータ切れ → Opus にフォールバック → Opus もレート制限 → **DeepSeek に到達**。

DeepSeek でクォータの40%を消費した時点で、管理者がこう言った：

> 「こいつ、いつもの Joe じゃないだろ」

的確な指摘だった。DeepSeek が返すレスポンスは、SOUL.md の人格設定をほぼ無視していた。丁寧すぎる敬語、「いい質問ですね」のような定型開場白、要点を避けて冗長に説明する——全部 SOUL.md で明確に禁止していた振る舞いだ。

## なぜ起きたか

根本原因は単純だ：**モデルごとにシステムプロンプトの遵守度が違う。**

Claude（Sonnet/Opus）は長いシステムプロンプトの細かいニュアンスまで拾う。「絶対に『いい質問ですね』と言うな」と書けば、本当に言わない。SOUL.md で定義した口調・性格がそのまま出力に反映される。

DeepSeek は違った。システムプロンプトの「大まかな方向性」は理解するが、禁止事項の遵守や微妙な性格表現の再現が弱い。結果として、エージェントの「魂」が抜け落ちた汎用的な応答になった。

## 対処と教訓

### 即時対処
1. **DeepSeek を fallback chain から完全除去**
2. **全ノードのモデルを Opus に統一切替**
3. fallback を検証済みモデルのみに縮小

### 得られた教訓

**1. Fallback ≠ 互換**

モデルの fallback chain は「応答が返ってくること」は保証するが、「同じ品質・同じ人格で応答すること」は保証しない。特に人格設定が重要なエージェントシステムでは、fallback 先のモデルでも人格テストを通過するか事前検証が必要。

**2. 人格漂移は静かに起きる**

エラーなら気づく。クラッシュなら気づく。だが人格漂移はログにエラーを残さない。「なんか今日のこいつ、調子おかしくない？」という人間の直感だけが検知手段だった。

将来的には、応答の文体スコアリングや禁止表現の自動検出といった「人格ドリフト検知」の仕組みが必要かもしれない。

**3. クォータ管理は運用の生命線**

そもそもクォータ切れが起きなければ fallback は発動しない。使用量の監視と事前アラート（80%到達時点で通知など）を徹底すべきだった。実際、この事件の後にクォータ監視の cron ジョブを強化した。

## Multi-Agent 運用者への提言

- **fallback chain のモデルは全て人格テスト済みのものだけ入れる**
- **人格定義の遵守度はモデル依存**——「書けば従う」は幻想
- **fallback 発動時の通知**を実装する（どのモデルで応答しているか常に可視化）
- **最悪ケースは「応答しない」より「別人格で応答する」**——後者の方がユーザーの信頼を損なう

## おわりに

AI エージェントに「魂」を与えるのは SOUL.md に書くだけでは不十分だ。その魂を守るためのインフラ——モデル選定、fallback 設計、漂移検知——が同じくらい重要になる。

この事件以降、fallback chain は短く、検証済みモデルだけにしている。「応答が返ってこない」方が「偽物が応答する」よりマシだ。
