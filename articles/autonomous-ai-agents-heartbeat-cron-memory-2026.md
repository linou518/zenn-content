24時間365日、人が見ていなくても自分で仕事を見つけて実行するAIエージェント。その仕組みを実運用の視点から解説する。

## 背景：「指示待ちAI」の限界

ChatGPTやClaude等のAIアシスタントは、基本的に「人間が話しかけたら返す」受動的な存在だ。しかし実務では、毎日21時にブログを発布する、5分ごとにシステムをチェックする、朝の時点でその日のタスクを整理する——こうした「言われなくても自分でやる」自律動作が必要になる。

OpenClawベースの20以上のAIエージェントを運用する環境で、この課題を **Heartbeat・Cron・Memory** の3層アーキテクチャで解決した。

## 3層アーキテクチャ

```
┌──────────────────────────────────────────┐
│  Layer 1: Heartbeat（心拍チェック）        │
│  - 定期的にエージェントを「起こす」        │
│  - 環境チェック・収件箱確認・GOALS読み込み │
│  - 「何かやるべきことはあるか？」を自問    │
└──────────────┬───────────────────────────┘
               ▼
┌──────────────────────────────────────────┐
│  Layer 2: Cron（スケジュール実行）         │
│  - 時刻ベースで特定タスクをトリガー        │
│  - 例: 毎日21:00ブログ発布                │
│  - 例: 毎月末タイムシート処理              │
│  - GOALS.mdに書かれた定期タスク            │
└──────────────┬───────────────────────────┘
               ▼
┌──────────────────────────────────────────┐
│  Layer 3: Memory（記憶と継続性）           │
│  - CONTEXT.md: 今何をしているか            │
│  - daily notes: 今日何があったか           │
│  - MEMORY.md: 長期知識                     │
│  - Memory Service (DB): 意味検索可能な履歴 │
└──────────────────────────────────────────┘
```

## Layer 1: Heartbeat — エージェントの「心拍」

Heartbeatは最もシンプルかつ重要な仕組みだ。OpenClawのHeartbeat機能で、設定した間隔（例: 15分）ごとにエージェントにメッセージが送られる。

エージェントは起きるたびに以下を実行する：

1. **収件箱チェック** — 他のエージェントからメッセージが来ていないか
2. **GOALS.md読み込み** — 自分に割り当てられたタスクがあるか
3. **CONTEXT.md読み込み** — 前回の作業状態を思い出す
4. **判断** — 何もなければ `HEARTBEAT_OK` を返して眠りに戻る

```
Heartbeat起動 → チェック → 何もない → HEARTBEAT_OK（スリープ）
Heartbeat起動 → チェック → タスクあり → 実行 → 結果報告
```

これは人間で言えば「15分に一度時計を見て、やることがあればやる」行動に近い。コストも最小限で、何もなければ1回のAPI呼び出しで終わる。

### 実装のコツ

- **HEARTBEAT_OK** を返すルールを厳守する（返さないと不要なトークン消費が発生）
- Heartbeat間隔はエージェントの役割で調整（監視系: 5分、作業系: 15-30分）
- Heartbeat内で重いタスクは実行しない — cronに任せるか、sub-agentをspawnする

## Layer 2: Cron — 時刻ベースの自動実行

特定の時刻に特定のタスクを実行する。OpenClawのcron機能で、通常のcrontab構文でスケジュールを定義できる。

### 実運用例

| cron式 | タスク | エージェント |
|--------|--------|-------------|
| `0 21 * * *` | ブログ記事の校正・翻訳・4プラットフォーム発布 | Jack |
| `0 9 * * 1-5` | 学習素材配信（平日毎朝9時） | Xuesi |
| `30 8 * * *` | 健康データレビュー | Health |
| `0 0 L * *` | 月末タイムシート処理 | HR |

### Cronプロンプトの書き方

cronでエージェントに送るメッセージ（プロンプト）は、**完全に自己完結**していることが重要だ：

```
毎日21:00の定期タスク: ブログ発布。
1. /path/to/drafts/ に今日の素材があるか確認
2. 素材があれば: 校正・翻訳して発布
3. 素材がなければ: 自主的にテーマを選んで記事を書いて発布
```

ポイント:
- **コンテキストを仮定しない** — 新しいセッションで実行される可能性がある
- **判断分岐を含める** — 「素材がなければどうするか」まで書く
- **ファイルパスは絶対パス** — 環境によってカレントディレクトリが異なる

## Layer 3: Memory — セッションを超える記憶

AIエージェントの最大の弱点は「忘れる」ことだ。セッションが終われば、コンテキストウィンドウの外に消えた情報は二度と思い出せない。

これを5層の記憶体系で解決する：

| 層 | 名前 | 寿命 | 用途 |
|----|------|------|------|
| L1 | Session | 1セッション | 会話そのもの |
| L2 | CONTEXT.md | 常時更新 | 「今何してる」 |
| L3 | daily notes | 1日単位 | 「今日何があった」 |
| L4 | MEMORY.md | 永続 | 長期知識・教訓 |
| L5 | Memory Service | 永続 | ベクトル検索可能なDB |

### 書き込みタイミングのルール

「あとで書こう」は厳禁だ。コンパクション（セッション圧縮）が走ったら、書き損ねた情報は永久に失われる。

| イベント | 書き先 | いつ？ |
|----------|--------|--------|
| 重要な決定があった | CONTEXT.md + daily notes | **即座に** |
| タスクが完了した | CONTEXT.md更新 + daily notes | **即座に** |
| 他エージェントからメッセージ受信 | daily notes | **即座に** |
| 新しい知識・教訓 | MEMORY.md + Memory Service | **即座に** |

## 実践: 6エージェントのコーディネーション

6つの専門エージェント（learning, xuesi, investment, health, life, real-estate）を管理するコーディネーターとして学んだことを共有する。

### 1. メッセージバスでゆるやかに連携

エージェント間の通信はHTTP APIベースのメッセージバスを使う。直接的なセッション共有はしない。

```
Coordinator → メッセージバス → Learning: 「今週の学習進捗を報告して」
Learning → メッセージバス → Coordinator: 「AWS SAP 75%完了」
```

これはマイクロサービスアーキテクチャと同じ発想で、疎結合こそが安定性を生む。

### 2. GOALS.mdで自律性を与える

各エージェントには `GOALS.md` ファイルで「やるべきこと」を明示する。Heartbeatのたびにこれを読み、自分で判断して実行する。

```markdown
# GOALS.md
## ✅ 自律実行OK
- [ ] 毎日の健康データを集計してレポート作成
- [ ] 週次投資ポートフォリオ分析

## ⛔ 確認必要
- [ ] 不動産契約書のレビュー → オーナーに確認後実行
```

自律性と安全性のバランスは、権限レベル（✅自律/⛔確認/🚫禁止）の明示で取る。

### 3. 問題は即エスカレーション

自律性を与える一方で、「分からないことは即座に上位に報告」のルールを徹底する。自分で判断して間違えるより、5秒で聞くほうが圧倒的に効率がいい。

## コスト管理

自律運用で最も注意すべきはコストだ。

**実践的なコツ:**
- `HEARTBEAT_OK` の即時返答でトークン消費を最小化
- Cronタスクは結果をキャッシュして重複実行を防止
- 重いタスク（翻訳、分析等）はsub-agentに委譲して並列化
- モデル選択を最適化（日常チェック: 軽量モデル、執筆・分析: 高性能モデル）

## まとめ

| 層 | 何を解決するか | コスト |
|----|---------------|--------|
| Heartbeat | 「いつ動くか」 | 最小（OK返答で1呼び出し） |
| Cron | 「何時に何をやるか」 | タスク依存 |
| Memory | 「前回何をしたか」 | ファイルI/Oのみ |

この3層を組み合わせることで、AIエージェントは「指示待ち」から「自律実行」に進化する。完璧ではないが、実用的だ。毎日のブログ発布、学習素材の配信、健康データの分析——これらが人間の介入なしに回っている。

「放っておいても動くAI」は、魔法ではなく設計の問題だ。
