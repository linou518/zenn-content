# ロールバック手順を現状だと思うな: 復旧作業で先に見るべきは runtime truth

運用では、「リポジトリに何があるか」「スクリプトに何が書いてあるか」「compose ファイルで何を宣言しているか」を、そのままシステムの現在状態だと思い込みがちです。

危険です。

本当に重要なのは、今この瞬間に何が実際に動いているかです。私はこの差を **source truth** と **runtime truth** の分離として捉えています。

最近の復旧作業でも、この問題がまた表に出ました。目的は本来、ある環境にカスタムプラグインを再配置することだけでした。しかし、デプロイスクリプト、イメージタグ、compose 設定だけを見ていると、「もう揃っている」と誤認しやすい。先に確認すべきなのは、むしろ次の点です。

- 今、本番でどのコンテナが実際に動いているか
- volume が誤った世代を引き継いでいないか
- プラグインはファイルとして存在するだけで、実際には有効化されていないのではないか
- 新環境の業務データが旧環境の内容で上書きされていないか
- 画面上で機能が見えない原因が、フロント入口なのか、バックエンドの設定なのか、プラグイン状態なのか

これらはすべて runtime truth の確認です。

## なぜ source truth は人を誤らせるのか

source truth は重要です。理想状態を定義する材料だからです。ただし、復旧・ロールバック・移行では、システムはすでに理想状態からズレていることが多い。

よくある誤解を挙げます。

### 1. コードがあるから機能もあるはず

違います。

プラグインのソースが repo にあるのは、「その機能を実装したことがある」ことの証明にすぎません。次のことは何も保証しません。

- プラグインがビルドされている
- ビルド成果物が正しいイメージに入っている
- そのイメージが正しく配備されている
- コンテナが新しい版で起動している
- プラットフォーム側でプラグインが有効になっている
- フロントエンドで入口が実際に見えている

どこか 1 か所でも切れれば、ユーザーに見える結果は「機能がない」です。

### 2. compose ファイルが正しいから本番も正しい

これも違います。

本番コンテナは再作成されていないかもしれない。古い volume が残っているかもしれない。環境変数がさらに前の版から来ているかもしれない。サービス名だけ合っていて、内部状態がズレていることもあります。

`docker compose config` が教えてくれるのは「どう起動されるべきか」です。`docker ps`、`docker inspect`、マウント先、プラットフォーム内の有効化状態が教えてくれるのが「今どう動いているか」です。

### 3. ロールバックスクリプトが走ったから復旧済み

これは特に危険です。

スクリプトが成功したのは、スクリプト自身の処理が終わったという意味でしかありません。業務状態が目標版に戻ったことは保証しません。

本当に確認すべきなのは次です。

- 重要データが期待どおり戻っているか
- 重要プラグインが最終ユーザーから見えているか
- 重要導線で実操作が 1 回通るか
- 新旧環境の境界が再び混線していないか

ここを見ていなければ、復旧完了は見かけだけです。

## 私が勧める復旧判断の順番

「デプロイを間違えたので戻す」という場面では、次の順で見るのが有効です。

### 1. まず保護対象を定義する

最初に明確にするべきなのは、何を守るのかです。

- **データ** を守るのか
- **プラグイン/コード** を守るのか
- **現環境のアイデンティティ**（ユーザー、agent、設定）を守るのか

多くの復旧事故は、技術不足ではなく、保護対象が最初に定義されていないことから起きます。プラグインだけ戻すつもりがデータまで触ってしまう。コード同期のつもりが、稼働中環境を上書きしてしまう。こういうズレです。

### 2. repo より先に runtime を見る

確認順序はこうです。

1. 本番のプロセス/コンテナ状態
2. volume / bind mount
3. プラットフォーム内の有効化状態
4. ユーザーに見える入口
5. 最後に repo、スクリプト、イメージ

この順番なら、「コードはあるのに」という錯覚に先に落ちません。

### 3. エンジニア視点ではなくユーザー導線で検証する

エンジニアは次で安心しがちです。

- ファイルがある
- API が 200 を返す
- コンテナが動いている
- ログにエラーがない

足りません。

有効なのは、ユーザー導線を実際に通すことです。

- 入口が見えるか
- クリックして開けるか
- 核心操作が完了するか
- 結果が期待どおりか

ユーザー導線が通らないなら、復旧は終わっていません。

### 4. 「部分復旧」を正式な状態として扱う

自動化では、部分成功は普通に起きます。

たとえば 4 プラットフォーム公開で 3 つ成功して 1 つ失敗する。プラグインは戻ったが token 失効で同期だけ失敗する。コード反映は終わったが、プラットフォームのスイッチだけ入っていない。

ここで最悪なのは、全部まとめて「失敗」とだけ記録し、何も残さないことです。

実務では次の方が有効です。

- 成功部分を記録する
- 失敗部分を明示する
- 再試行可能な状態を残す
- 次回は失敗した箇所だけを補う

復旧も公開も、現実には二値ではなく段階的収束です。

## すぐ使える確認チェックリスト

復旧やロールバックをするとき、私は先にこの 6 つを確認するようにしています。

1. 守る対象はコードか、データか、環境アイデンティティか
2. 今見ているのは source truth か runtime truth か
3. コンテナ、volume、環境変数、プラットフォーム設定はそれぞれどうなっているか
4. ユーザー視点で対象機能は見えて、使えるか
5. どこまでが本当に成功で、どこからが見かけだけの成功か
6. 失敗時に再試行情報を残したか、それとも最初からやり直しになるか

難しい話ではありませんが、これだけでかなりの初歩的事故を防げます。

## まとめ

ロールバックや復旧で一番危険なのは、エラーが出ることではありません。**エラーが出ていないのに安心すること**です。

スクリプトが完了した、コンテナが動いている、repo のコードが正しい。どれもシステム状態の正しさを保証しません。先に信じるべきなのは runtime truth、つまり今この瞬間に本番で何が動き、ユーザーが何を使えるかです。

ここを確かめない限り、復旧完了という判断は、問題をもっと見えにくい場所へ移しただけかもしれません。
