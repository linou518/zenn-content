20以上のAIエージェントが動く環境で、「あの話いつしたっけ？」に答えられるシステムをゼロから作った記録。

## 背景：エージェントは忘れる

OpenClawベースで20以上のAI agentを運用している。各agentは独立したセッションで動き、コンテキストウィンドウを超えた過去の会話は事実上消える。

マークダウンファイル（daily notes, MEMORY.md）で記憶を補完していたが、限界がある：

- **検索がテキストマッチのみ** — 「あの投資の話」で「不動産利回り計算」がヒットしない
- **agent間の情報共有が困難** — agent Aで話した内容をagent Bが知らない
- **22,000件超の会話履歴** — grepでは無理

## アーキテクチャ

```
┌─────────────────────────────────────────────┐
│  Session Sync Daemon                        │
│  - 5分ごとにOpenClaw全セッションをDBに同期   │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│  PostgreSQL + pgvector                      │
│  - sessions / messages テーブル              │
│  - embedding列 (vector(1536))               │
│  - 22,778メッセージ, 748セッション           │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│  Memory Service (FastAPI)                   │
│  - /search: ハイブリッド検索(意味+テキスト)  │
│  - /facts: 構造化された事実の保存・検索       │
│  - /messages: 生メッセージ取得               │
│  - Access Log付き（全クエリ記録）            │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│  Embedding Backfill Worker                  │
│  - OpenAI text-embedding-3-small            │
│  - 全22,778件を非同期でベクトル化            │
│  - 新規メッセージは同期時に自動embedding     │
└─────────────────────────────────────────────┘
```

**技術スタック**: Python (FastAPI) / PostgreSQL 16 + pgvector / OpenAI Embeddings / systemd

## 実装で学んだこと

### 1. Export方式 vs DB直接検索

最初は「DBの内容を30分ごとにMarkdownにexportして、既存のファイル検索（memory_search）で読む」方式を採用していた。

**問題点**：
- exportはagent単位でファイルを生成 → メッセージ単位の精度が出ない
- 30分のラグがある
- agent間の横断検索ができない

**決断**: exportを停止し、DB直接検索に一本化。ロールバックは設定を戻すだけなのでリスクも低い。

結果、検索精度と即時性が大幅に向上。冗長なexport cronも消えてシンプルになった。

### 2. 「使われているか分からない」問題

DBに切り替えた直後、管理者から「で、これ実際にagentが使ってるの？」と聞かれた。

正直、分からなかった。curlで叩くだけなので、agentが自主的に呼ばなければログがない。

**対策を2つ実装**：

**A. Access Log** — FastAPIにミドルウェアを追加し、全APIコールをjournalctlに記録。`journalctl --user -u memory-service | grep SEARCH`で誰がいつ何を検索したか一発で分かる。

**B. memory_db Skill** — OpenClawのスキルとして登録し、agentの会話ログにcurl実行が表示されるようにした。「使われている」ことが対話の中で自然に見える。

### 3. Embedding Backfill — 全量ベクトル化の威力

22,778件のメッセージを全てembeddingした。`text-embedding-3-small`で1536次元。

backfill前は新規メッセージしかベクトル検索できず、「先週の話」が意味検索で引っかからなかった。backfill完了後、過去の全会話が意味的に検索可能に。

コスト的にも`text-embedding-3-small`なら22,000件程度は数ドルで済む。

### 4. セッション同期のタイミング

5分間隔の同期で十分だった。リアルタイム性が必要な場面はほぼなく、「さっき話したこと」は同一セッション内のコンテキストで拾える。5分前の別セッションの話が必要になるケースは稀。

## 運用して気づいたこと

- **双査モード**が実用的 — ファイルベースの`memory_search`とDB検索を併用する。ファイルにしかない情報（手書きメモ、設定ファイル）もあるので、完全にDB移行はしない
- **agent_id フィルタ**で個別agent検索と全体横断検索を切り替えられるのが便利
- **PostgreSQL接続プール**の"rolling back returned connection"ログが30秒ごとに出て一瞬焦ったが、psycopg_poolの正常動作だった（実害なし）

## まとめ

20+ agentの記憶を統合管理するために：

1. PostgreSQL + pgvectorでセッションDBを構築
2. 5分間隔の自動同期でOpenClawの全会話を取り込み
3. 全量embeddingで過去の会話も意味検索可能に
4. Access Logで利用状況を可視化
5. ファイル検索との双査モードで網羅性を確保

「AIエージェントに記憶を持たせる」問題は、結局のところ普通のデータベース設計の問題だった。pgvectorのおかげでベクトル検索も同じDB内で完結する。特別なベクトルDBは不要。
