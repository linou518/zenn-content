Dashboard の予定表示を見直していて、価値を生んでいるのは Outlook 連携そのものではなく、**生の予定データをそのまま返さず、サーバー側で「今日の実行ブロック」に再編集して返していること**だとあらためて感じました。

Techsfree の Dashboard には、予定データの入口として `tasks_ms.json` があります。ここには会議や休憩などの raw なスケジュール情報が入っています。しかし、この形のままフロントエンドに渡しても、ユーザーがその日をどう回すかまでは分かりません。会議一覧は見えても、「会議の前後に何をやるのか」「空き時間をどう使うのか」は別のロジックが必要です。

実際に画面が見ているのは、`schedule/schedule.json` のように整形された block データです。そこでは会議だけでなく、深度工作、PR確認、営業数字確認、小休憩のような作業単位まで含めて、`start`、`end`、`label`、`type`、`status` を持つ時系列ブロックとして返しています。つまり API の責務を「イベント一覧を渡す」から「その日を運転できる粒度に整形する」へ移しているわけです。

この差は見た目以上に大きいです。もしフロント側で raw meetings と tasks を都度マージするなら、会議衝突判定、空き時間の埋め方、休憩の扱い、表示順保証、集中ブロックの差し込みといったロジックが UI 層に流れ込みます。すると、単なる表示調整のつもりが、実質的にはスケジューラ本体を触ることになります。

サーバー側で block 化しておくと、フロントエンドはかなり単純になります。表示コンポーネントは「何をどう計算したか」を知らなくてよく、配列を順に描画すればよい。これは単なる責務分離ではなく、**壊れ方を限定する設計**です。スケジュール生成の不具合は API 側で閉じ、表示崩れは UI 側で閉じる。障害切り分けが速くなります。

また、raw データに少し怪しい値が混じっても、最終出力を block に正規化できる点も強いです。たとえば開始時刻と終了時刻が同じ会議データのような、扱いに迷う入力があっても、最終的な 1 日の block 列として整えることで UI へ生の揺れを直接持ち込まずに済みます。

SaaS 連携では「まず API から取れたものを画面に出す」段階で満足しがちですが、運用で本当に価値が出るのはその一歩先です。特に複数案件を並行で回す Dashboard では、必要なのはカレンダーの忠実な再現ではなく、**次の30分を決めやすい形への再編集**です。だったらフロントで頑張るより、サーバーで block を生成してしまったほうが長持ちします。

結論は明快です。**予定 API は「イベント一覧」を返すより、「行動単位のタイムブロック」を返したほうが強い。** 見た目が同じ予定表でも、設計の重心をどこに置くかで、保守性も使い勝手も大きく変わります。
