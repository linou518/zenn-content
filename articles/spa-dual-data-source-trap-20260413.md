ダッシュボードの「プロジェクトタスク」画面を見直していて、いかにもフロントエンドらしい罠に当たった。症状は単純で、タスクを追加した直後は画面に見えるのに、リロードすると消える。最初に疑うべきは API エラーや保存失敗だが、今回はそこではなかった。原因はもっと嫌なやつで、**同じ画面に見えて実装が二重化していた**ことだった。

整理すると、フロントには2系統の実装が混在していた。ひとつは `app.js` 側で、`tasks.json` を `loadData()` で読み込み、追加・削除・完了切替を `/api/task/add` `/api/task/delete` `/api/task/toggle` に流す構成。こちらはバックエンド経由なので永続化される。もうひとつは `index.html` の内聯スクリプトで、`simpleProjectsData` というメモリ上のオブジェクトを直接書き換えるだけの構成。画面上では同じ「タスクが増えた」ように見えるが、実際には後者はブラウザ内の一時状態を触っているだけなので、更新後に消える。

この手の不具合が厄介なのは、UI だけ見ると「一応動いている」ように見える点だ。追加ボタンも反応するし、一覧も更新される。だから最初は API やファイル保存より、描画のほうに目が向きやすい。でも実際には、**状態の正本が二つある時点で設計負け**している。片方は `tasks.json`、もう片方はページ内メモリ。これでは整合性が崩れるのは時間の問題だった。

今回やったことは派手ではない。まず、バックエンドの `/api/task/add` が `title` だけでなく `task` も受けられるようにして、画面側から自然に呼べるようにした。次に `/api/task/delete` を追加し、対応する Markdown から該当行を消したうえで `_regenerate()` して `tasks.json` を再生成する流れにした。つまり「見た目を変える」ではなく、「保存経路を一つに寄せる」を優先した。

学びは明快で、SPA の不具合は見た目より**状態管理の所有権**を疑ったほうが早い、ということだ。特に、単一ページを長く運用していると、応急処置のスクリプトや旧実装が残って、同じ責務を別ルートで持ち始める。そうなると、バグ修正の本質は if 文追加ではなく、どこを唯一の真実にするかを決め直すことになる。

フロントエンドの仕事は、きれいな見た目を作ることより、ユーザーの操作結果が時間をまたいでも裏切られないようにすることだと思う。ボタンが動くことと、本当に機能していることは別。今日の修正は、その当たり前をもう一度思い出させるタイプの作業だった。
