20以上のAIエージェントが互いに通信する仕組み。軽量メッセージバスの設計思想と実運用の知見を共有する。

## 問題：エージェント同士はどう会話するか？

AIアシスタントが1つだけなら、通信は問題にならない。人間とAIが直接対話すれば済む。しかしエージェントが10を超え、複数サーバーに分散すると、根本的な課題が浮上する：**エージェント間の通信をどう実現するか？**

筆者の環境では20以上のエージェントが9ノードに分散し、それぞれ異なる領域（業務プロジェクト、健康管理、投資、教育など）を担当している。エージェント間で頻繁に発生する通信パターンは以下の通り。

- **タスク委譲**：統括エージェントがサブタスクを専門エージェントに割り振る
- **状態同期**：あるエージェントがタスク完了後に他のエージェントへ通知する
- **情報リクエスト**：Agent Aが Agent Bの持つ知識を問い合わせる
- **ブロードキャスト**：システムレベルの告知（メンテナンス、ルール更新など）

## 方案選定：なぜ既製MQを使わないのか

最初に思いつくのはRabbitMQ、Redis Pub/Sub、NATSだろう。しかしAIエージェントのユースケースでは、これらは**オーバースペック**だ。

AIエージェントの通信特性：
- メッセージ頻度が低い（1日数十〜数百件、毎秒数万件ではない）
- メッセージ本体が小さい（テキスト指示が中心）
- **受信箱セマンティクス**が必要（オフライン中のメッセージを保持し、復帰後に取得）
- シンプルな**送受信者識別**で十分

これらの特性から、最もシンプルな構成を選択した：**HTTP APIサービス + SQLiteストレージ**。

## アーキテクチャ

```
┌───────────┐    HTTP POST     ┌────────────────┐
│ Agent A   │ ────────────────→│                │
└───────────┘  /api/send        │  Message Bus   │
                                │  (Node.js)     │
┌───────────┐    HTTP GET       │                │
│ Agent B   │ ←────────────────│  SQLite DB     │
└───────────┘  /api/inbox/:id   └────────────────┘
```

コアコンセプトは3つだけ。

1. **送信** — `POST /api/send` で from/to/subject/body を指定
2. **受信** — `GET /api/inbox/:agentId` で未読メッセージを取得
3. **ブロードキャスト** — toを `ALL` にすれば全エージェントへ配信

### API設計

```bash
# メッセージ送信
curl -X POST http://bus-server:8091/api/send \
  -H "Content-Type: application/json" \
  -H "X-Bus-Token: <TOKEN>" \
  -d '{"from":"joe","to":"jack","subject":"タスク割当","body":"本日のblog発布を頼む"}'

# 受信箱取得（既読マーク付き）
curl http://bus-server:8091/api/inbox/jack?mark_read=true \
  -H "X-Bus-Token: <TOKEN>"

# オンラインエージェント一覧
curl http://bus-server:8091/api/agents \
  -H "X-Bus-Token: <TOKEN>"
```

### 認証

最もシンプルな静的Token認証を採用。全エージェントが共通Tokenを `X-Bus-Token` ヘッダーで送信する。

なぜ複雑な認証を使わないか？メッセージバスは内部ネットワーク上で動作し、エージェントはすべて信頼済みだからだ。セキュリティ境界はネットワーク層（内部ネットワーク分離）であり、アプリケーション層ではない。

## 実運用での設計判断

### 1. 受信箱モデル vs リアルタイムPush

**受信箱モデル（Pull型）** を選択した理由：

- AIエージェントの「オンライン」は間欠的 — Heartbeatは5分間隔、Cronは特定時刻にトリガー
- リアルタイム応答は不要。次回起動時に受信箱をチェックすれば十分
- Pull型は本質的に冪等で、切断再接続のハンドリングが不要

実運用では、各エージェントがセッション開始時にまず `受信箱チェック` を行う。

### 2. メッセージフォーマット：構造化だが過剰にしない

```json
{
  "from": "joe",
  "to": "jack",
  "subject": "Blog発布リマインド",
  "body": "素材をdrafts/に配置済み。21:00に発布フローを実行してくれ",
  "timestamp": "2026-04-03T12:00:00Z"
}
```

メールのfrom/to/subject/bodyの4要素を踏襲。直感的で十分。優先度、タグ、スレッドなどの複雑な概念は導入しない — エージェントにGmailの機能は不要だ。

### 3. ブロードキャストの実装

`to: "ALL"` の場合、メッセージバスが登録済み全エージェントの受信箱にメッセージをコピーする。シンプルかつ実効的。

実際のユースケース：メンテナンス前に統括エージェントが「15分後にinfraノードを再起動する。作業状態を保存せよ」とブロードキャスト。

### 4. メッセージの永続化

SQLiteで全メッセージを保存。未読メッセージは取得されるまで保持、既読メッセージは7日後に自動削除。

SQLiteを選んだ理由：
- 単一ファイルでバックアップが容易
- 同時書き込み量が極めて少なく、PostgreSQLは不要
- クエリ能力は十分（エージェント別フィルタ、時間範囲検索）

## 実際の活用パターン

### パターン1：タスク委譲チェーン

```
Linou → Joe(統括) → Jack(blog) → 完了通知 → Joe → Linou報告
```

Joeがユーザーの指示を受け、メッセージバス経由で専門エージェントにサブタスクを振る。完了報告もバス経由。

### パターン2：エージェント間情報照会

```
投資Agent: "学習agent、最近の量的取引関連のノートある？"
学習Agent: "先週Factor IC/IRの整理をした。パス: /shared/knowledge/quant/..."
```

### パターン3：定期ステータスレポート

各エージェントがHeartbeat時に自身のステータスサマリを統括エージェントに送信。統括エージェントが集約してダッシュボードを更新。

## 踏んだ地雷

### 地雷1：メッセージ消失

初期は永続化なしで、メッセージバス再起動時に全未読メッセージが消失した。SQLite導入で解決。

**教訓**：AIエージェントにとって、メッセージ消失は遅延より致命的 — エージェントは「受け取ったはず」を覚えていない。

### 地雷2：ブロードキャストストーム

デバッグ中にあるエージェントがループ内でブロードキャストを連発し、20以上の受信箱が一瞬で溢れた。同一送信者あたり毎分10件のレートリミットを追加。

### 地雷3：単一障害点

メッセージバスが固定サーバー上で動作しているため、そのサーバーのメンテナンス時に全エージェント間通信が途絶。

現在の対策：エージェントがメッセージバス到達不能を検知した場合、「ローカルのみモード」にフォールバックし、クロスエージェント通信に依存しないタスクは継続実行する。

## 外部通信チャネルとの関係

メッセージバスはエージェント間の内部通信を担う。人間との通信は別チャネル。

```
人間 ←→ Telegram/Mattermost ←→ エージェント
エージェント ←→ メッセージバス ←→ エージェント
```

この分離は重要だ。内部通信は外部サービスを経由せず、高速かつ高信頼。内部の会話が外部プラットフォームに漏れることもない。

## 今後の発展可能性

現在の方案で十分機能しているが、将来の進化方向として：

1. **メッセージ優先度**：緊急メッセージ（セキュリティアラートなど）はエージェントを即座に起動すべき
2. **処理確認メカニズム**：送信側が受信側の処理完了を把握できる仕組み
3. **エージェント自動発見**：新エージェントのオンライン時に自動登録

ただしYAGNI原則に従い、実際に必要になるまで実装しない。

## まとめ

AIエージェント間通信の核心的なニーズは、**シンプルで信頼性の高い非同期メッセージングだ**。HTTP API + SQLiteの軽量構成は、どのエンタープライズMQよりもこのユースケースに適している。

設計原則：
- **PullがPushに勝る** — エージェントの間欠的オンライン特性に適合
- **受信箱セマンティクス** — メッセージの消失なし・重複なし
- **最小複雑度** — メール4要素フォーマットで十分
- **ネットワーク層セキュリティ** — 内部ネットワーク信頼環境では複雑な認証は不要

20以上のエージェントがこの仕組みで約2ヶ月安定稼働。1日平均200件のメッセージ処理でゼロロス。時に、最もシンプルな解が最良の解である。
