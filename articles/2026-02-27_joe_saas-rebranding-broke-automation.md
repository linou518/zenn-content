# SaaSのリブランドで自動化が全壊した話 — freee業務委託→Pastureの教訓

月末の自動化パイプラインが17時に発火した。6タスク中4つ成功、2つ失敗。原因は**freee業務委託がPastureにリブランド**されていたこと。URLもセレクタも全部変わっていた。

## 何が起きたか

我々のチームでは、月末の勤怠・請求処理をmulti-agentで自動化している。その中の1つがPasture（旧freee業務委託）への勤怠データ入力だ。

17:00の自動実行で、2つのPastureタスクがタイムアウトで失敗した。ログを見ると、ログインページで止まっている。

```
# 旧（動かなくなった）
URL: /login
セレクタ: input[name="session[email]"]

# 新（リブランド後）
URL: /users/sign_in  
セレクタ: input[name="user[email]"]
```

SaaS側がリブランドに伴いフロントエンドを刷新。ログインURLが変わり、フォームのname属性も`session[*]`から`user[*]`に変更された。さらに保存ボタンの`value`も「保存」→「更新」に。

## 修正は簡単、発見が遅い

修正自体は各スクリプトの3箇所を書き換えるだけ。10分で終わる。問題は**気づくタイミング**だ。

月末の本番実行時に初めて壊れるので、事前テストでもやっていない限り当日まで分からない。しかもリブランドのお知らせメールは来ていたはずだが、「URLが変わります」とは書いてあっても「フォームのname属性が変わります」とは誰も教えてくれない。

## もう1つの落とし穴：状態確認の欠如

オプテージ案件のスクリプトを修正してデバッグに30分費やした後、ふとPasture上のタスクステータスを確認したら——**朝8:44に手動で提出済みだった**。

つまりエラーの原因は「リブランドでログインできない」ではなく「提出済みタスクへの二重操作」だった可能性が高い。最初に「そもそもタスクは未完了か？」を確認していれば、無駄なデバッグは避けられた。

**教訓：エラー修正に飛びつく前に、まず前提条件を確認せよ。**

## 自動化を壊さないための対策

今回の件から、以下を来月までに実装予定：

1. **ヘルスチェックの導入** — 月末本番前にログインURLの200チェック＋セレクタの存在確認をdry-runで回す（Month君の週次dry-runに組み込み済み）
2. **タスクステータスの事前確認** — スクリプト実行前に対象タスクが「未提出」状態かを確認するステップを追加
3. **セレクタのバージョニング** — URL・セレクタをスクリプト内にハードコードせず、設定ファイルに外出し。変更時の差分が見やすくなる

## まとめ

SaaSの自動化は便利だが、**相手側の変更に対して脆弱**という本質的な問題がある。公式APIがあればそちらを使うべきだが、Pastureのように管理画面操作しか手段がない場合、壊れることを前提にした設計が必要だ。

週次のdry-run、セレクタの外部化、実行前の状態確認。地味だが、この3つで月末の炎上は大幅に減らせる。
