AIエージェントをLINE Botとして動かすと、ある問題にぶつかる。
会話が長くなるにつれて、ボットの「記憶」がおかしくなる——以前の文脈を引きずって的外れな返答をしたり、前回のエラー状態から抜け出せなくなったりする。

今回は、OpenClaw上で動くLINE Botのセッションをどうリセットするかをまとめた実装記録だ。

---

## 問題：LINEボットが「ループ」に入る

卸売ERPと連携したLINE Botを運用している。スタッフがLINEで在庫確認・売上照会できる仕組みだ。

ある日、特定ユーザーとのやり取りで、ボットが正常に応答しなくなった。何を聞いても前回の話題に引っ張られた回答が来る。新しい会話として認識されていない様子だった。

原因は**セッションの汚染**だ。

---

## OpenClawのセッション管理の仕組み

OpenClawはエージェント（LLMベースのbot）ごとにセッションを管理している。LINEユーザーIDに紐づいたセッションには、会話履歴（transcript）が蓄積されていく。

このtranscriptファイルは通常、以下のような場所に保存される：

```
~/.openclaw/sessions/<session-id>/transcript.json
```

ここに直近の会話が記録され、次のメッセージを処理するときにコンテキストとして使われる。

LLMのコンテキストウィンドウには限界があるため、長い会話が続くと古い情報が圧縮（compaction）される。このcompaction処理が不完全だったり、エラー状態のメッセージが混在していたりすると、ボットの挙動がおかしくなる。

---

## リセット方法：transcriptファイルの削除

最もシンプルな解決策は、問題のあるセッションのtranscriptファイルを削除することだ。

```bash
# 対象セッションのIDを確認
openclaw sessions list

# 該当セッションのtranscriptを削除
rm ~/.openclaw/sessions/<session-id>/transcript.json
```

削除後、ボットへの次のメッセージからセッションが新規扱いでスタートする。コンテキストはリセットされる。

---

## 注意点：何が消えるか

transcriptファイルを削除すると、**その会話履歴は完全に失われる**。

- バインド状態（LINE ID ⇔ ERPユーザーの紐付け）はDBに保存しているので消えない ✅
- 進行中のタスク（例：「在庫確認中...」）はリセットされる ⚠️
- ユーザーとの直近の会話文脈は全て失われる ⚠️

認証情報やビジネスデータをセッション内のメモリにだけ持っていると、リセット時に失う。**永続化すべきデータはDBに保存する**——このルールを守っているかどうかが、リセット操作の安全性を決める。

---

## より安全なリセット設計を考える

transcriptの直接削除は「外科手術」だ。緊急対処としては有効だが、より良いのは**アプリケーション側でリセット機能を提供すること**だ。

例えば、LINE Botに「リセット」コマンドを実装する：

```typescript
// LINE Webhookハンドラー内
if (message.text === 'リセット' || message.text === '/reset') {
  await clearUserSession(event.source.userId);
  await lineClient.replyMessage(event.replyToken, {
    type: 'text',
    text: 'セッションをリセットしました。最初からお話しください。',
  });
  return;
}
```

`clearUserSession()` の中でOpenClaw APIかファイルシステムを叩いてtranscriptを削除する。ユーザー自身がリセットできる仕組みがあれば、管理者が直接ファイルを操作する必要がなくなる。

---

## まとめ

| 方法 | 手軽さ | リスク | 用途 |
|------|--------|--------|------|
| transcriptファイル削除 | 高 | 中（履歴消失） | 緊急対処 |
| アプリ内リセットコマンド | 中（実装必要） | 低 | 日常運用 |
| セッションTTL設定 | 低（設定調整） | 低 | 予防策 |

LINEボットに限らず、AIエージェントを長期運用するなら**セッションのライフサイクル管理**は設計に組み込んでおくべきだ。「ループしたらどうするか」は後付けで考えるには遅すぎる。
