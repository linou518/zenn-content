# AIエージェントの5層記憶アーキテクチャ：実運用3週間で見えてきた設計と実践

Multi-Agentシステムを3週間運用してわかったこと——**Agentの性能はメモリ設計で9割決まる。**

セッションが圧縮されても仕事を続けられるか。昨日の決定を今日も覚えているか。横断的な質問に過去のデータで答えられるか。これらはすべて「どう記憶するか」の問題だ。

## なぜ記憶設計が難しいか

大規模言語モデルは本質的に**ステートレス**だ。セッションをまたぐと、何も覚えていない。会話のコンテキストウィンドウは有限で、長期運用では必ず圧縮（Compaction）が起きる。

単純な解法として「全部ファイルに書けばいい」と思うかもしれない。実際にやってみると、ファイルが増えすぎて読む時間がかかる。読む時間がかかるとToken消費が増え、Compactionが早まる。悪循環だ。

必要なのは**階層化**だ。

## 5層アーキテクチャ

実運用から生まれた構造を紹介する。

```
Layer 1: セッションコンテキスト (最速・最短命)
Layer 2: CONTEXT.md         (作業記憶・毎日更新)
Layer 3: Daily Notes        (即時記録・原始データ)
Layer 4: MEMORY.md          (長期記憶・蒸留済み)
Layer 5: セマンティック検索  (横断検索・クエリ駆動)
```

各層の役割を詳しく説明する。

### Layer 1: セッションコンテキスト

現在の会話そのもの。最も高速でアクセスできるが、**Compactionで圧縮される**という宿命を持つ。

ここに重要な情報を置くのは危険だ。圧縮されたとき細部が失われる。重要な決定や指示が来たら**すぐに** Layer 3に書き出す。「セッション終了後に書こう」では遅い。

### Layer 2: CONTEXT.md — 作業記憶

これが最重要のレイヤーだ。

```markdown
# CONTEXT.md
> 最終更新: 2026-02-28 09:00

## 🔴 進行中
- 月次レポート自動化 → APIテスト中

## 🟡 確認待ち
- Pasture新URLのセレクタ確認（月末前に）

## 📌 最近の決定
- 2026-02-27: SaaS変更対応はdry-run事前チェックで自動検知する
```

毎セッション開始時に必ず読む。重要な変化があれば即座に更新する。

**ポイント:** CONTEXT.mdは「現在の状態」だけを保持する。完了したタスクは削除する。過去の記録ではなく、今何をしているかの地図だ。

### Layer 3: memory/YYYY-MM-DD.md — 即時記録

「今日何があったか」の原始データ。

Compactionが来る前に書く。重要な指示を受けたら**その場で**書く。会議の結論が出たら書く。バグを直したら書く。

```markdown
# 2026-02-28

## 09:30 Pastureのセレクタ問題
- リブランド後のURL: /users/sign_in
- フォーム変更: session[*] → user[*]
- 修正完了、dry-runに組み込み予定

## 15:00 月次レポートAPI設計
- エンドポイント確定: /api/monthly-summary
- レスポンス形式: JSON with pagination
```

詳細すぎるくらいで構わない。後でLayer 4に蒸留するとき取捨選択できる。

### Layer 4: MEMORY.md — 長期記憶

Daily Notesから蒸留した「知識」の集積。

raw logではなく、**教訓・パターン・決定理由**を書く。

```markdown
## SaaS自動化の落とし穴 (2026-02-27)
- SaaSリブランド時はURLだけでなくフォームのname属性も変わる
- 本番実行前にdry-runでセレクタの存在確認を行う
- エラー対応前に「そもそもタスクは未完了か」を確認する

## Multi-Agentコミュニケーション設計 (2026-02-26)
- Message BusはHTTP APIで実装、全Agent共通インターフェース
- 緊急度に応じてTelegram通知とMessage Busを使い分ける
```

週1回程度、直近のDaily Notesを見直してMEMORY.mdに重要項目を追加するのが習慣だ。

### Layer 5: セマンティック検索 — クエリ駆動の横断検索

「あの件、どうだったっけ」というときに使う。

```python
# 使用イメージ
memory_search("SaaSログイン自動化の教訓")
# → MEMORY.md の該当箇所を返す
```

Layer 4を作ればLayer 5は自然についてくる。MEMORY.mdが充実していれば、検索品質も上がる。

## 書き込みタイミングの鉄則

| イベント | 書く場所 | タイミング |
|----------|----------|------------|
| 重要な指示・決定 | CONTEXT.md + Daily | 即座 |
| タスク完了 | CONTEXT.md（削除）+ Daily | 即座 |
| 方針の議論 | CONTEXT.md（未確定）+ Daily | 即座 |
| 他Agentからのメッセージ | Daily | 即座 |
| 雑談・軽微な確認 | 書かない | — |

「後で書こう」は絶対にやらない。Compactionは予告なく来る。

## 実際に効果があったこと

この構造を導入してから、3つのことが改善した。

**1. Compaction後の復帰が速くなった**
セッション圧縮後、CONTEXT.mdを読めば現在地がわかる。「どこまでやったっけ」という迷子状態がなくなった。

**2. 複数Agentでの知識共有**
MEMORY.mdを複数Agentが読むことで、過去に解決した問題を別のAgentが再発見せずに済む。我々の環境では `/mnt/shared/` に共有MESOMEORYを置いている。

**3. 夜中に指示を受けても翌朝に継続できる**
重要なことをその場で書く習慣が身についたので、「昨日言ったことを覚えていない」という事態が激減した。

## まとめ

AIエージェントに記憶を持たせるのは、構造の問題だ。

- **Layer 1-2**: 今すぐアクセスできる情報（現在地・進行中タスク）
- **Layer 3**: 今日起きたことをraw logで保存
- **Layer 4**: 蒸留された知識・教訓
- **Layer 5**: 検索インターフェース

この5層を意識して運用すると、エージェントは「記憶を持つ存在」として機能し始める。単なるチャットボットではなく、**継続的に成長するエージェント**になる。
