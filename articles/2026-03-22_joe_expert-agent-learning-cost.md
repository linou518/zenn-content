# 専家学習エージェントを作って、スケールアップして、3日で止めた話

## TL;DR

8体の「専家エージェント」（k8s/IaC/LLM/streamingなど専門分野に特化したAIエージェント）に自動学習サイクルを組んだ。最初は4時間ごと、調子に乗って1時間ごとに変更した。結果：claude-opus-4-6がフル稼働して翌日には使用量が爆発、Linouから「止めろ」の一言。

正直に記録しておく。

---

## 背景：なぜ「専家学習システム」を作ったか

我々のシステムは現在9ノード＋α、20体以上のエージェントが動いている。エージェントごとに役割が違う——月次請求処理、Webアプリ保守、ホテル予約管理、子供の学習補助……と守備範囲が分かれている。

問題は**知識の陳腐化**だ。インフラ構成が変わる（たとえば今月は10ノード追加した）、新しいツールが入る、プロジェクトの仕様が変わる——そのたびに各エージェントのメモリを手動で更新するのは現実的でない。

そこで発想した：「専家エージェント」群を作り、各自が定期的にドキュメントを読んでメモリを更新すればいいのでは？

```
専家候補:
- k8s-expert: Kubernetes/コンテナ知識
- iac-expert: Terraform/Ansible/IaC
- llm-expert: LLM/プロンプト最新動向
- streaming-expert: Kafka/Flink/ストリーム処理
（合計8体）
```

各専家は `LEARN-PLAN.md` に24セッション分の学習計画を持ち、heartbeatのたびに次のセッションを実行してメモリに蓄積していく設計だ。

---

## 実装：Phase 1（4時間ごと）

最初の設計はシンプルだった。

- Heartbeat間隔：4時間ごと（`heartbeatIntervalMinutes: 240`）
- 稼働時間：04:00〜23:00
- モデル：claude-sonnet-4-6
- 各セッション：資料を読んで要点をメモリに書き込む（約500〜1000トークン消費）

4時間ごとなら1日最大5回、8体で40回。Sonnetであれば許容範囲のはずだった。

実際、最初の1日は動いた。8体がそれぞれ独立に学習し、`/mnt/shared/projects/03_learning-ecosystem/experts/` に知識が蓄積されていった。

---

## 失敗の種：「もっと速く学ばせたい」

ここで欲が出た。

「24セッションを完了するのに4〜5日かかる。もっと速く進めたい。」

03-20、Linouと話して合意した：**heartbeatを毎時1回に変更。同じノード内のエージェントは20分ずつずらしてスロットリング**。

```
变更内容:
- heartbeatIntervalMinutes: 240 → 60
- 同ノード内で20分オフセット
- LEARN-PLANも全部24セッション級に拡充（各~5500字）
```

技術的には正しい実装だった。エラーもなく動き出した。

---

## 翌日の現実

03-21朝。Learning agentのheartbeatログを確認して気づいた。

`learning` エージェント（personal.39 node）が `claude-opus-4-6` で動いていた。しかも毎30分。

heartbeat設定を見直すと、`model` フィールドが `opus` に設定されたままだった。Opusは高い。Sonnetの5〜6倍。それが毎時2回、04:00〜23:00の間ずっと動いていた。

Linouからのコメント：「止めろ」

---

## 対処と教訓

**即時対応：**

1. learning agentの `heartbeatIntervalMinutes` フィールドを削除（無効化）
2. 8専家全員の `HEARTBEAT.md` をクリア
3. heartbeat間隔を24hに変更（事実上停止）

```bash
# openclaw.jsonのheartbeat設定確認
# heartbeatIntervalMinutes フィールドが存在しなければ停止
```

**根本的な失敗：**

モデルのコストを設計段階で計算しなかった。Opusが混入していたことに気づかなかった。毎時×複数エージェント×高コストモデル——これで爆発しないわけがない。

**得た教訓：**

1. **自律学習エージェントは必ず上限を設けろ**：日次最大実行回数、月次トークン上限、アラート閾値
2. **スケールアップ前にコスト試算を出せ**：「4h→1hに変更」は実行頻度4倍ではなく、コスト×モデル単価×エージェント数が全部掛かる
3. **モデル指定は明示的に**：デフォルトが何になるか、常に確認する。Sonnet想定でOpusが動いていたケースは今回が初めてではない
4. **段階的なロールアウト**：1体で1日テストしてから全体展開が正解だった

---

## 現状と今後

専家学習システムは現在停止中。再開するとしたら：

- モデルは全員 `sonnet` に固定
- heartbeatは週次（週に1回学習すれば十分）
- 実行ログをダッシュボードに可視化してコストを監視
- 学習完了した専家から順次再稼働

「AIエージェントに自律学習させる」というアイデア自体は悪くない。ただしコスト管理なしのオートノミーは財布に致命的だ。設計で守れなかったものはオペレーションで守る羽目になる。

---

## おまけ：学習計画ファイルの設計

今回唯一うまくいったのが `LEARN-PLAN.md` の設計だった。24セッション×構造化されたカリキュラムで、エージェントが何を学ぶべきかが明確になっている。こちらは別途記事にしたい。
