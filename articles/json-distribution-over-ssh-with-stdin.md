今日の Dashboard 改修では、複数ノードに `auth-profiles.json` を配る処理を見直しました。もともとは API キーを含む JSON を base64 化し、`ssh ... "echo '<base64>' | base64 -d > auth-profiles.json"` のように流していました。見た目は手軽ですが、運用ではここが思った以上に脆いです。JSON が長い、改行が入る、シェルの引用が絡む、ノードごとの差が出る――こうした条件が重なると、ときどき壊れます。そして**ときどき壊れる処理**が一番厄介です。

今回の修正は単純で、リモート側では `cat > target` だけを実行し、JSON 本文は `subprocess.run(..., input=auth_content, text=True)` で stdin からそのまま渡す形に変更しました。ローカルノードでは Python で直接ファイルを書き、リモートノードだけ SSH を通します。ポイントは、**JSON をシェル文字列として扱わない**ことです。

base64 は便利ですが、シェルをまたぐ瞬間の quoting 問題を完全には消せません。最初から stdin 転送に寄せたほうが、実装もデバッグも素直になります。

副作用として、処理の責務も整理できました。`set_subscription` 側は「どのキーをどのノードへ配るか」だけを考え、実際の書き込みは `write_auth_profiles()` に閉じ込めました。こうすると失敗点を単独で追いやすくなり、将来 `auth-profiles.json` の構造が変わっても修正箇所を一か所に絞れます。SaaS や API 連携では、通信そのものよりも**設定をどう安全に配るか**で事故ることが多いので、この分離は地味ですが効きます。

学びは二つあります。ひとつ目は、秘密情報を含む設定ファイルを shell one-liner の上に載せないこと。ふたつ目は、複数ノード運用では「たまたま動く」より「壊れ方が単純」な実装を選ぶことです。派手な改善ではありませんが、こういう土台の修正はあとで効きます。UI を足す前に、まず配布の足場を固める。結局それが一番速いです。
