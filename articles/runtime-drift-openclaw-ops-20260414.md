# ソースにあるのに動かない: OpenClaw運用で runtime drift を先に疑うべき理由

OpenClaw 周辺の運用で厄介なのは、ソースコードと実際の runtime が静かにズレることです。今回ハマったのは、AI Back Office Pack の workflow 画面が「押しても反応しない」ように見えた事象でした。最初は UI の不具合や API エラーを疑いたくなりますが、根因はもっと単純でした。**ソースには workflow があるのに、動いている Docker コンテナが古いままだった**のです。

こういう障害は、コードを読んだだけでは見誤ります。リポジトリ上には workflow 関連の実装があり、画面側にも部品がある。すると人は「実装済みなのに動かない。認可かルーティングか」と考えがちです。でも運用で最初に確認すべきなのはそこではありません。**今動いている artifact が、本当にそのソースから build されたものか**です。

今回の切り分けは、source → build artifact → running container → route → auth の順で進めました。この順序を守るだけで、無駄な深掘りをかなり減らせます。ソースに対象実装があることを確認したあと、次に見るべきは build 成果物にそのコードが含まれているかどうかです。ここを飛ばして API ログやブラウザコンソールに潜ると、平気で 30 分から 1 時間溶けます。実際、今回も稼働中の `api` と `dashboard` コンテナが旧イメージのまま残っており、新しい workflow モジュールを持っていませんでした。

つまり、見えていた現象は「機能不全」ではなく「未配備」でした。ソース上の truth と runtime 上の truth が一致していなかったわけです。Docker を使っていると、compose 定義を更新した、ソースを pull した、手元で dist が生成された、という事実だけで安心しがちです。でも本当に重要なのは、**今 listen しているプロセスが、どのイメージ・どの build 成果物で起動しているか**を確認することです。

対応自体は単純でした。`ai-backoffice-pack` の `api` と `dashboard` コンテナを rebuild / recreate し、workflow を含む実行物へ差し替えるだけで、「なぜか反応しない」状態はそのまま消えました。ここで得た教訓は二つあります。ひとつは、ソース確認だけで安心しないこと。もうひとつは、障害の切り分け順序を固定することです。

OpenClaw のように、AI、Web、ジョブ、認証、コンテナがつながる運用では、人は複雑な原因を探しに行きがちです。でも実際には、古いコンテナ、古い dist、反映されていない設定といった地味なズレが根因であることが多い。派手な原因から探すと負けます。

自分の中では、今回で確認順がかなり明文化されました。**「ソースにある」で止めず、「そのコードが build され、いま動いているプロセスに載っている」と言えるまで確認する。** これを飛ばすと、運用は簡単に嘘をつきます。AI システムでも、最後に信用すべきなのは説明ではなく runtime です。

## すぐ使える確認順

1. ソースに対象実装が存在するか確認する
2. build artifact に成果物が含まれているか確認する
3. running container 内に期待したファイルやモジュールがあるか確認する
4. route が存在するか、応答が 404 / 401 / 500 のどれかで切り分ける
5. 最後に auth・permission・frontend の順で深掘りする

「コードはあるのに動かない」と感じたら、まず application bug ではなく **runtime drift** を疑ったほうが早いです。
