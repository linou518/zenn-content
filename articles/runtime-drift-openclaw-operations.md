4月8日に扱った不具合の中で、いちばん実務的だったのが、**ソース上には機能があるのに本番では動かない**という典型例でした。対象は `ai-backoffice-pack` の workflow 機能です。

現象だけ見ると、月末 workflow の管理画面が反応しません。最初はフロント実装漏れや API ルーティング漏れを疑いたくなりますが、実際に確認すると、`dashboard/src/pages/Workflow.tsx` は存在し、backend 側にも `backend/src/modules/workflow/` がありました。つまり**機能そのものはコード上に存在していた**わけです。

それでも `/api/v1/workflows/steps/definitions` を叩くと `Route not found` が返る。ここで見るべき対象はソースではなく、**今動いている実体**です。調べると、稼働中 API コンテナの `dist/modules` に workflow モジュールが入っていませんでした。原因はコード不足ではなく、**古いイメージのままのコンテナが生き残っていたこと**でした。これが runtime drift です。開発者は「コードはある」と考え、利用者は「画面が壊れている」と感じる。その間で、実行中の実体だけが過去に取り残されます。

対処は地味ですが確実です。infra ノードで `docker compose build api dashboard` を実行し、続けて `docker compose up -d api dashboard` で再作成しました。重要なのは確認方法で、単に「コンテナが起動した」で終わらせないことです。`/app/dist/modules/workflow` が存在すること、さらに workflow definitions エンドポイントが 404 ではなく 401 を返すことまで確認しました。401 は未認証なだけで、少なくとも**ルート自体は正しく公開されている**と判断できます。ここまで確認して初めて「直った」と言えます。

この件で改めて効いたのは、Docker 化された業務アプリの切り分け順でした。

1. ソースに機能があるか
2. build artifact に入っているか
3. 稼働中コンテナに入っているか
4. ルートが公開されているか
5. 認証を越えた先で正しく動くか

1 だけ見て安心すると、かなりの確率でハマります。逆に 3 と 4 まで見ると、故障位置が一気に狭まります。

同じ日にもう一つ大きかったのが、会計システム統合の判断です。既存の別システムをそのまま API 連携する案は捨て、画面とアップロード体験だけを参考にしつつ、freee 連携ロジックは `freee-bookkeeper` から引き抜き、最終的に `ai-backoffice-pack` の backend / dashboard / Postgres に統合する方針にしました。ここでも重要なのは、「今動いている別システムがある」ことと、「その運用構成を増やすべきか」は別問題だという点です。短期の再利用と長期の保守負債を混ぜると、あとで確実に重くなります。

実運用では、**ソース、ビルド成果物、コンテナ、公開ルート、認証後レスポンス**までそろって、ようやく同じ機能と言えます。runtime drift は派手な障害ではありませんが、放置すると一番時間を奪うタイプの不整合です。コードを疑う前に、今動いているものを疑う。これが今回の教訓でした。
