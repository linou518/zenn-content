# マルチプラットフォーム配信は全成功だけを正解にしてはいけない。部分完了を扱えるべきだ

自動配信を設計するとき、多くの人は「一度で全媒体に出す」ことを理想にします。これは間違いではありません。ただし、**100% の全成功しか認めない設計**は、実運用ではかなり弱いです。

なぜなら、マルチプラットフォーム配信は最初から単一点の処理ではなく、**入力は共通、出口は分散、故障面も独立**した仕事だからです。

同じ記事でも、Zenn は git push で通るかもしれません。Qiita は token scope に依存します。dev.to は API が受け入れる body 形式に左右されます。Hashnode は GraphQL mutation と publication 設定が揃って初めて通ります。テーマと内容と公開時刻は共有していても、**失敗の仕方までは共有していません。**

だから、1媒体の失敗で全体を中断し、最後に「publish failed」とだけ返す仕組みは、まだ理想環境の発想です。

## 本当に追うべきなのは全成功ではなく、説明可能な最大完了度

より現実的な目標は次です。

**内容の整合性を保ったまま、可能な媒体をできるだけ完了し、失敗は明示的に残す。**

これは失敗を甘く見る話ではありません。重要なのは、局所的な失敗が全体結果を飲み込まないことです。

典型例を挙げます。

- Zenn 成功
- Hashnode 成功
- dev.to 成功
- Qiita は token 失効で 401

このとき最悪なのは、実行全体を failed とだけ記録して終わることです。

業務上の実態は「全部失敗」ではありません。大半の外部分配は完了していて、残っているのは認証に限定された単一プラットフォームの問題です。

この差を表現できないと、運用側は誤った絵を見ます。表示は単なる失敗でも、実態は「4件中3件完了、残り1件は認証修正待ち」です。

## マルチプラットフォーム処理は媒体ごとに独立決算すべき

安定した構成は、「1本のコマンドで4媒体まとめて賭ける」形ではなく、次の流れです。

1. まず共通のコンテンツ成果物を作る
2. そのあと媒体ごとに個別送信する
3. 媒体ごとに結果を独立記録する
4. 最後に構造化された集計を出す

この形にははっきり利点があります。

## 1. 1媒体の異常が他の成功を消さない

Qiita が失敗しても、Zenn と dev.to がすでに公開済みなら、その成功は成功として残すべきです。最後の1ステップでエラーが出たからといって、全体を未実施扱いにしてはいけません。

## 2. 障害箇所の特定が速い

「publish failed」にはほとんど価値がありません。

「Qiita: 401 Unauthorized、Zenn: success、Hashnode: success、dev.to: success」なら価値があります。見るだけで、まず認証を直すべきだと分かるからです。内容、ネットワーク、全体フローを疑う必要はありません。

## 3. 再実行範囲を小さくできる

結果が構造化されていれば、次回は失敗した媒体だけを再試行できます。

これはリクエスト数の節約だけではありません。二重投稿、重複 commit、重複通知を防ぐ意味でも重要です。

## 自動化が怖いのは失敗そのものではなく、失敗が不透明なこと

配信系タスクでは、「自動で実行できること」を主目標にして、「結果の監査性」をおまけ扱いすることがよくあります。

実際には逆です。

cron で動く配信処理で重要なのは、スクリプトが走ったかどうかより、終わった直後に第三者が次を答えられることです。

- 今日はどの記事を出したか
- どの媒体へ出たか
- どこが失敗したか
- 失敗原因は権限か、形式か、ネットワークか、レート制限か
- 成果物と結果記録はどこにあるか

この問いにすぐ答えられないなら、半分成功していても運用としては未成熟です。

## 結果をログの副産物ではなく、一等成果物として扱う

信頼できる配信フローは、記事そのもの以外にも、安定して次の2種類を残すべきです。

- **再確認できる本文ファイル**：中国語原稿、日本語稿、英語稿
- **振り返れる結果ファイル**：媒体別状態、URL、HTTP code、失敗理由

つまり、結果は端末のスクロール出力だけに置くべきではありません。明示的なファイルとして残すべきです。たとえば：

- `publish-result.json`
- `publish-report.md`

前者は機械用、後者は人間用です。

こうしておけば翌日に見直すとき、shell history を掘り返したり、「昨日はどこで止まったのか」を推測したりする必要がありません。証拠が成果物ディレクトリに残っています。

## 私が見る実用的な判断基準

今の私は、マルチプラットフォーム配信の出来を次の一文で見ています。

**4媒体のうち1つが一時的に壊れていても、残り3つを完了し、その壊れた点を明確に残せるか。**

答えが No なら、その仕組みは自動配信ではなく、単なる直列の運試しです。

現実の自動化は、環境が常に完璧であることを前提にしてはいけません。局所失敗を受け止め、全体進捗を守り、次の修正に必要な情報を十分に残せるべきです。

マルチプラットフォーム配信で本当に価値があるのは、一度で全部通す能力ではありません。**全部通らない日でも、結果が整理され、状態が見え、あとから補えること**です。
